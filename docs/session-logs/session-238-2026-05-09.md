### Visual diagnosis via screenshot

Took a screenshot at z19 + tilt=48° (`/tmp/wedge-before-z19-tilt48.png`). The wedge is dark `#2D1B4E` at the very top corners (expected — outside the trapezoid, mathematically unfillable). The middle band between vanishing-point line and trapezoid top edge IS being painted by MapView's extended rows, but at z19 the perspective compression crushes street/building detail to the point where the rendered output reads as flat gray. At z16 the same compression preserves enough variation per pixel to look "filled."

So the operator's complaint isn't a tile-fetch / clipping / layout bug — it's a perceptual problem: extreme perspective compression × z19 tile detail = unrecognizable. Real 3D maps avoid this with LOD (lower-zoom tiles for distance) or fog/horizon overlays.

Computed the math at 48° tilt, current `topY = h * 1.0 * t`:
- Vanishing point at dst y=84 (inside visible screen) → top 12% of screen mathematically unfillable.
- Trapezoid top edge at dst y=364 → covers only 47% of screen.
- Band y=84..364 is the perceptual-failure band.

### Phase 3 fix shipped — `topY = h * 0.6f * t`

One-constant change to `TiltContainer.rebuildMatrix()`:

```kotlin
val topY = h * 0.6f * t   // was h * 1.0f * t
```

At 48°, vanishing point moves to dst y=-258 (off-screen top), trapezoid top moves to dst y=219, trapezoid covers ~68% of screen. Entire screen now theoretically fillable; perspective compression per pixel reduced ⇒ z19 detail stays legible.

Built, deployed, drove device via adb input (zoom + tilt + walk-sim teleport to NPS Visitor Center). Comparison screenshots:

- `/tmp/wedge-before-z19-tilt48.png` — pre-S238: upper ~50% featureless gray.
- `/tmp/wedge-z20-tilt48.png` — post-S238: full Salem detail (buildings, streets, road labels: Sumner Court, Briggs St, George Pope House, Andrew Safford House, etc.) right up to the small unfillable corner triangles.

Operator's stated requirement ("see a map that fills the screen") met.

### Cull boundary loosened (S237 carry-forward step 5)

`TiltContainer.drawBillboardedMarkers`:
```kotlin
// before: if (ctnY < 0f || ctnY > ch || ctnX < 0f || ctnX > cw) continue
// after:
if (ctnY < mvTop - 50f || ctnY > mvTop + mv.height + 50f ||
    ctnX < -50f || ctnX > cw + 50f) continue
```

Pre-S238, the cull rejected `ctnY < 0` because the wedge was largely unfilled and markers floated on `#2D1B4E`. With the wedge now mostly filled by S238's softened perspective, allow markers anywhere in MapView's extended drawing rect (-1710..+2394 in container coords). The post-matrix screen-bounds cull (`dx/dy < -300 || > +300+slack`) handles off-screen rejection.

Same loosening mirrored in `SpriteOverlay.drawBillboarded` — `ctnH = mapView.height` replaced with `mvBottom = mvTopPx + mapView.height` and the cull conditioned on `mvTopPx - 50 .. mvBottom + 50`.
