## THIRD PIVOT — Verbose persistent logging

Operator: *"Any kind of verbose logging we can add to the application so we have a good log for capturing issues?"*

### Why this matters

The pre-S110 `DebugLogger` was a 1,000-entry in-memory ring buffer with logcat bridge and optional TCP streamer. **All in memory.** When the process died at 16:38 today, every log entry from before that point was lost. The 14-minute black hole in the bug timeline is the direct consequence — we couldn't see what happened during the hang because nothing survived.

### Logging additions

**1. `DebugLogger.FileLogSink`** — daily-rotated, line-flushed file at `/sdcard/Android/data/com.example.wickedsalemwitchcitytour/files/logs/debug-YYYYMMDD.log`. Single background thread (`Executors.newSingleThreadExecutor`), `BufferedWriter` with `flush()` after every line. 7-day retention pruned on rotation. Wired in `WickedSalemApp.onCreate()` via `DebugLogger.initFileSink(applicationContext)` BEFORE any other init logs so the very first log line of the process is captured.

**2. `GPS-OBS` per-fix telemetry** — every fix the activity observes gets:
```
I/GPS-OBS: fix lat=42.55673 lng=-70.87159 acc=3.4m speed=0.00mps bearing=? gap=4997ms
```
With explicit `gap=` measurement of the time between consecutive fix arrivals.

**3. `GPS-OBS` heartbeat coroutine** — ticks every 10s. Logs in three cases:
- State transitions (healthy → stale, stale → recovered, no-fix → first-fix): immediate
- Periodic OK heartbeat: once per 60s while healthy (so post-mortem reader can confirm coroutine is alive)
- Steady-state stale: throttled to once per 30s while held in stale state

**4. `LIFECYCLE` observer** — logs `onResume`, `onPause`, `onStop`, `onDestroy` with the activity's `isChangingConfigurations` and `isFinishing` flags. Lets post-mortems distinguish "process death" (no graceful onPause/onDestroy) from "configuration change" (`changingConfig=true`) from "user backgrounded the app" (neither flag).

### Logging quality issues found mid-session

After installing the file sink + heartbeat, an audit revealed:

| Issue | Fix |
|---|---|
| **Heartbeat had logged ZERO lines ever** — by design it only logged on transitions, but with continuously healthy GPS the coroutine looked dead | Rewrote with periodic OK heartbeat every 60s + start marker `HEARTBEAT START — tick=10s, stale-threshold=30s, ok-log=60s` |
| **Sequence numbers reset on process restart** — `[53]` could refer to different events from different process instances when they were appended to the same daily file | Added PID to every line prefix: `[N\|PID]` format. Single grep `\|2315]` filters to one process |
| **`bearing=?°`** — degree sign appended even when bearing was null on stationary fixes | Drop unit suffix when value is null. Same fix to `speed=?` |
| **No process-start marker** — only date-rotation header existed, only fired on calendar day change | `DebugLogger.initFileSink()` now logs `════════ process started PID=N ════════` as the first entry of every fresh process |
| **930 historical stack trace lines in the file** | All from PID 26950 (the FIRST install of the day, pre-noise-tweak). New process runs are clean. |

## FOURTH PIVOT — Cache-proxy circuit breaker (A + B)

After the verbose logging landed, operator asked: *"are we seeing errors right now? If we can't connect locally to our server, it should not crash or affect the app."*

Audit found that when the cache-proxy at `10.0.0.229:4300` is unreachable (which it was — laptop on different IP / not running), every consumer (`WeatherViewModel`, `MainViewModel`, `GeofenceViewModel`, etc.) was independently:
- Hitting a 15-second TCP connect timeout
- Logging a 16+ line stack trace
- Retrying on every fresh trigger

Operator chose options A + B from the four proposed improvements:
- **(A)** Compact-mode logging for `ConnectException` to localhost/LAN — log a single line, no stack trace, when the cause is "we know this server isn't running."
- **(B)** Shared `LocalServerHealth` singleton with a circuit breaker: first failure flips it to `DOWN` for 60 seconds; consumers fail fast without even attempting TCP.

### Implementation

**New files in `core/util/network/`:**

| File | Purpose |
|---|---|
| `LocalServerHealth.kt` | Process-wide circuit breaker singleton. State: single `lastFailureAtMs: AtomicLong`. Methods: `isUp()`, `lastFailureAgeMs()`, `recordFailure(host, cause)`, `recordSuccess(host)`. Constants: `OPEN_DURATION_MS = 60_000L`, `LOCAL_HOSTS = setOf("10.0.0.229", "localhost", "127.0.0.1")`. Logs the OPEN/CLOSED transitions but stays quiet between them. |
| `CircuitOpenException.kt` | Sentinel `IOException` thrown by the interceptor when the circuit is open. Subclasses `IOException` so call sites that already catch it for transport failures handle it transparently. Recognized by `DebugLogger.e()` as a known network condition. |
| `LocalServerCircuitBreakerInterceptor.kt` | OkHttp `Interceptor`. Only acts on requests whose host is in `LocalServerHealth.LOCAL_HOSTS`; everything else passes through unchanged. If circuit OPEN → throw `CircuitOpenException` immediately. On `ConnectException` / `SocketTimeoutException` / `UnknownHostException` from chain → `recordFailure` + rethrow. On any returned `Response` (even 4xx/5xx — server is reachable even if unhappy) → `recordSuccess`. Includes `OkHttpClient.Builder.withLocalServerCircuitBreaker()` extension helper. |

**Wired into 13 `OkHttpClient.Builder` sites:**

12 repository files in `core/data/repository/`:
- `WeatherRepository`, `MbtaRepository` (both have a pre-existing per-API-key/User-Agent interceptor — added the circuit breaker BEFORE the existing interceptor in the chain)
- `TfrRepository`, `WebcamRepository`, `AircraftRepository`, `FindRepository`, `PlacesRepository`, `GeofenceRepository`, `GeofenceDatabaseRepository`, `ChatRepository`, `AuthRepository`, `CommentRepository`

1 inline OkHttp client in `app-salem`:
- `SalemMainActivityFind.kt` — the geocode client at `10.0.0.229:3000` (Nominatim port — different port than cache-proxy but same host, breaker tracks by host so it covers this too)

**`DebugLogger.e()` updated** to detect `ConnectException`, `SocketTimeoutException`, `UnknownHostException`, `CircuitOpenException` (and their cause chain via 5-level walk) and log a single compact line at WARN level instead of an ERROR-level message + stack trace. Real exceptions still get the full stack trace so unexpected bugs aren't hidden.

## FIFTH PIVOT — POI proximity encounter tracker

Operator: *"When we get close to POI's — I want us to track how close for how long (max/min distance and time) so I can review that later."*

### Design

- **Proximity radius:** 100m (narrower than the manager's 300m nearby radius, wider than the typical 30-50m geofence entry radius)
- **Per encounter:** poi_id, poi_name, poi_lat, poi_lng, first_seen_at_ms, last_seen_at_ms, duration_ms, min_distance_m, max_distance_m, fix_count
- **Lifecycle:** encounter opens on first GPS fix within 100m, updates running stats on every subsequent fix, closes when the POI leaves 100m
- **Persistence:** `poi_encounters` table in `UserDataDatabase` (alongside `gps_track_points`). Composite string PK `${poiId}|${firstSeenAtMs}` for `INSERT OR REPLACE` semantics without auto-id juggling across coroutine boundaries. DB version bumped 1 → 2 with `fallbackToDestructiveMigration` (rolling 24h GPS journey is acceptable to lose on schema upgrade).
- **Retention:** 7 days, pruned hourly (mirrors `GpsTrackRecorder` pattern).
- **Threading:** `recordObservation()` runs on the main thread (called from the GPS observer); the in-memory `active` map is touched only there. DB writes dispatched to IO scope and never touch `active`. No locking required.
