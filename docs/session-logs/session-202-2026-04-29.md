## Recon Camera architecture (V1)

Three iterations driven by operator feedback during the session:

### Iteration 1 — system camera via `ACTION_IMAGE_CAPTURE`

- Manifest: added `<uses-permission android:name="android.permission.CAMERA" />` + `<uses-feature android:required="false">`.
- `res/xml/file_paths.xml`: added `<external-files-path name="recon_camera_tmp" path="recon-camera-tmp/" />`.
- New `KatrinaCameraManager.kt` orchestrator: lazy-bound FusedLocationClient + ROTATION_VECTOR sensor; `requestCapture()` → permission check → launch `MediaStore.ACTION_IMAGE_CAPTURE` with FileProvider URI temp file; `onCaptureResult()` → `ExifInterface` writes lat/lon/alt/speed/track/img-direction/timestamp/processing-method/UserComment, then publishes to MediaStore `Pictures/WickedSalem-Recon/` (RELATIVE_PATH on Q+; IS_PENDING transition).
- New menu item `menu_top_camera_recon` on the Material toolbar with `ic_camera_recon.xml` (solid white photo-camera glyph, distinct from the stroked `ic_camera` used for "CAMs"/webcams).
- Wired ActivityResultLaunchers in `SalemMainActivity` (capture + permission); registered launchers via `katrinaCameraManager.registerLaunchers()`; sensor lifecycle hooked in onResume/onPause.
- `MenuEventListener.onCameraReconRequested()` added to the core interface with default `onStubAction("camera_recon")` so the generic-app implementer compiles without override.
- Build added `androidx.exifinterface:exifinterface:1.3.7` to app-salem dependencies.

**First install crashed.** Crash buffer caught it: `KatrinaCameraManager.<init>` at line 57 NPE on `activity.applicationContext` — the manager was constructed during `SalemMainActivity` field initializers (before `super.onCreate()`), at which point the activity's base context isn't yet attached. Fix: `by lazy` on every context-dependent field (`context`, `sensorManager`, `rotationVector`, `fusedClient`).

### Iteration 2 — moved to slim toolbar

Operator: "I was expecting the camera button to be on the top menu next to the home icon." The Material toolbar overflow vs. the slim toolbar's directly-placed ImageViews are two different surfaces. Moved:

- Added `<ImageView id="@+id/toolbarCameraIcon">` to `toolbar_two_row.xml` between Home and Weather (8dp marginStart matches the Weather margin).
- `AppBarMenuManager.setupSlimToolbar()` got a `cameraIcon: ImageView?` parameter; click wires to `menuEventListener.onCameraReconRequested()`.
- `SalemMainActivity.onCreate()` passes the ref via `findViewById(R.id.toolbarCameraIcon)`.
- Removed `menu_top_camera_recon` from `menu_main_toolbar.xml` and the dispatch from `AppBarMenuManager.kt:102` when() — eliminated the duplicate surface.

### Iteration 3 — in-app CameraX preview activity

Operator: "I need an easy way to leave the camera with an X — can we make this a submenu or something internal so we don't lose control on the window?" Right call: system camera handoff is opaque; OEM camera apps vary in how they exit. Built our own.

- Added CameraX 1.3.4 deps (core / camera2 / lifecycle / view).
- New `ReconCaptureActivity.kt`: full-screen Activity, declared in manifest with `screenOrientation="portrait"` and `Theme.AppCompat.NoActionBar`. Reads `EXTRA_OUT_FILE_PATH` from intent. Binds `Preview` + `ImageCapture` (CAPTURE_MODE_MAXIMIZE_QUALITY, DEFAULT_BACK_CAMERA). FLAG_KEEP_SCREEN_ON during framing.
- Layout `activity_recon_capture.xml`: `PreviewView` filling parent; top-left **X close** in a 56dp translucent dark circle (`recon_chip_bg.xml` + `ic_recon_close.xml`); top-center HUD line for status text; bottom-center 84dp shutter button (`recon_shutter_button.xml` — white circle with thin gap inside ring).
- Back button maps to the same cancel path as the X.
- `KatrinaCameraManager.launchCameraIntent()` rewritten to launch `ReconCaptureActivity` instead of `ACTION_IMAGE_CAPTURE`. Same temp-file pattern: manager creates the path, activity writes the JPEG, manager runs EXIF + MediaStore + cleanup on RESULT_OK.
