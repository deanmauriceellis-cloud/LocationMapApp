## Splash-skip investigation (open)

Operator reported: tapping the launcher showed only the system-icon splash (~0.5 s "witchykitty picture") then jumped straight to the GPS map.

Logcat evidence (17:39:25 launch):
- `TransitionRequestInfo type=TO_FRONT triggerTask=taskId=476 baseIntent=SalemMainActivity`
- `onTaskMovedToFront taskId=476, baseIntent=SalemMainActivity`
- `Start proc 4531 for next-top-activity {SalemMainActivity}`
- Zero `SplashActivity:` log tags for this process (`WickedSalemApp`/`SplashVoice`/`SalemMainActivity` all present).

Root cause: SplashActivity and SalemMainActivity share the default task affinity. Splash `finish()`es itself after launching the main activity (`SplashActivity.kt:143`), so the task root becomes SalemMainActivity. Next launcher tap matches on affinity and Android brings the existing task to front rather than routing through the LAUNCHER intent-filter activity.

Verified fix path via `adb shell am force-stop && am start -n .../SplashActivity`: splash plays the full Katrina intro + TTS welcome (`speaking — utteranceId=splash_welcome`).

Proposed fix (awaiting operator go-ahead): add `android:launchMode="singleInstance"` to SplashActivity in `app-salem/src/main/AndroidManifest.xml`. Splash would live in its own task; SalemMainActivity auto-forwards to a separate default-affinity task; subsequent launcher taps always land on a fresh SplashActivity.

## Narration silence bug (found from logcat)

Operator reported "I am not hearing narration" right after the install. Logcat showed:

```
I NARR-PLAY: DIRECT PLAY: Bewitched Sculpture ... bodyLen=749
W NarrationMgr: TTS not ready — skipping: poi_narration_1616966265342376
I TextToSpeech: Connected to TTS engine   (7 ms too late)
```

First POI geofence ENTRY fires ~7 ms before `NarrationManager`'s TTS engine binds. `enqueue()` was dropping any segment when `!ttsReady`. The OLD dual-clip path masked the race because the body clip landed a few ms later with TTS ready. The NEW single-clip path had no fallback, so the first POI went silent.

Fix in `app-salem/.../tour/NarrationManager.kt`:
- `enqueue()` no longer drops on `!ttsReady` — runs gates, appends to queue, logs `I TTS not ready yet — deferring ${segment.id}`, returns.
- `onInit()` TTS-ready callback now calls `playNext()` if `queue.isNotEmpty() && _state.value is NarrationState.Idle` — flushes whatever piled up during the bind window.

Rebuilt + clean-reinstalled. Operator confirmed narration audible.
