## Data archaeology — critical finding

Investigated `salem-schema.sql` and `tools/salem-data/`. Salem POI data lives in **three places**:
- `salem_tour_pois` (PostgreSQL + Room) — ~45 curated tour stops
- `salem_businesses` (PostgreSQL + Room) — ~860 curated + scraped
- **`narration_points` (Room ONLY, generated by Python pipeline)** — ~814 POIs from `tools/salem-data/narration-priority-pois.json`. **NOT in PostgreSQL.** Generated by `tools/salem-data/generate_narration_sql.py` writing into `salem-content/salem_content.sql`. The 814 narration points are almost certainly where the user's "position errors" and "overlapping icons" complaints originate (Overpass scrape provenance).

This forced an architectural decision before any coding: where does the canonical editable POI data live going forward?

## Decisions confirmed (AskUserQuestion dialog)

1. **Q1 Data foundation:** Migrate `narration_points` to PostgreSQL (new `salem_narration_points` table). One-time importer. PostgreSQL becomes single source of truth.
2. **Q2 Auth:** HTTP Basic Auth via env vars (`ADMIN_USER` / `ADMIN_PASS` in `cache-proxy/.env`). Browser handles native prompt.
3. **Q3 Categories:** Hand-port `PoiCategories.kt` into TypeScript config in `web/src/config/`. Add cross-reference comments in both files. Unification deferred.
4. **Q4 Data flow:** PostgreSQL is canonical post-migration. JSON files in `tools/salem-data/` retire to historical artifact status. Edits flow: admin tool → PostgreSQL → "Publish" regenerates `salem_content.sql` → next APK ships fresh data.
