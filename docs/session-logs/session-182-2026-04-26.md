### Investigation — tour rendering vs P2P

Code paths read:

- `app-salem/.../tour/WalkingDirections.kt` — both `getRoute(from, to)` (P2P) and `getMultiStopRoute(waypoints)` (tour) call into the same `Router` via `SalemRouterProvider`.
- `routing-jvm/.../routing/Router.kt` — `route()` is single-pair Dijkstra. `routeMulti(stops)` is literally `for i in 0..stops.size-2: route(stops[i], stops[i+1])` followed by `concat(parts)`. No simplification, no smoothing, no separate algorithm.
- `concat()` — appends each leg's geometry, skipping the first point of every leg after the first to dedupe the join. Since `nearestWalkableNode(lat, lng)` is deterministic, leg N ends at and leg N+1 starts at the same snap-node lat/lng; the join is geometrically continuous.
- `routeResultToWalkingRoute()` — adds straight-line "approach segments" for `leadingApproachFrom` (first stop) and `trailingApproachTo` (last stop) only, gated by `APPROACH_MIN_M=8` / `APPROACH_MAX_M=60`. Intermediate tour stops get no approach segments — the polyline at each intermediate stop sits at its snap-node lat/lng (on the graph), not at the POI lat/lng.
- `app-salem/.../ui/TourViewModel.kt:246` — `computeTourPolyline()` builds the waypoint list from `activeTour.stops` resolving each `stop.poiId` to its POI's `lat,lng`. Then calls `getMultiStopRoute(points)`. No pre-snap, no waypoint adjustment.
- `app-salem/.../ui/SalemMainActivityTour.kt:983` — `drawTourRoute()` takes the returned `route.polyline` and hands it directly to OSMdroid `Polyline.setPoints()`. No simplification on the rendering side either.

**Verdict on the explore-agent "snap-divergence between consecutive legs" theory: false.** Snap is deterministic — `nearestWalkableNode(B)` returns the same node when called as the "to" of leg N or the "from" of leg N+1. Concat de-dupes that node correctly. There is no "jump between adjacent snap nodes that interpolates straight through a building."

**Verdict on tour-rendering being broken vs P2P being intact: there is no code-path divergence.** Tour rendering = N P2P calls + concat. The polyline for tour leg `i→i+1` is identical to what `route(stops[i], stops[i+1])` would produce as a P2P call, byte-for-byte after the join-dedupe.

What that means for the operator's complaint:

1. P2P "Samantha → Old Town Hall" (using current GPS, not the Samantha POI position, as the start) routed cleanly. That's one specific Dijkstra path — and it doesn't traverse any building-crossing edge.
2. The Heritage Trail tour traverses 9 leg paths, each its own Dijkstra. At least one of those legs traverses a `salem.edges` row whose stored polyline geometry visually intersects a historic-building polygon.
3. S181 PostGIS analysis already identified that bad-data set: 14 walkable `salem.edges` whose `the_geom` cuts a `salem-historic-buildings.geojson` polygon by more than 1 m, with the worst cluster (Salem Green / St Peter St / City Hall Ave) on the Heritage Trail corridor.

So the visual artifact is in the bundled `salem.edges` geometry, not in any tour-specific code path. The 9-leg tour traverses some of those edges; the specific P2P pair the operator tested doesn't.

**Options to fix (no work done yet):**

A. **Surgical edge-geometry edits.** UPDATE the 14 bad rows in `salem.edges` to route around the buildings (re-shape the LineString, keep the same source/target nodes), re-bake the routing bundle, rebuild AAB. Smallest blast radius. ~1 session.

B. **Drop the 14 bad edges from the walkable subset** (`walkable=false`) and let Dijkstra route around them. Some legs lengthen modestly. Risk: legs that depend on those edges as the only path become longer or unrouted. Fastest to ship. ~half session.

C. **Switch routing input from TIGER to MassDOT Roads** for V2. Different schema, full re-bake of the graph. Right answer long-term, too big under V1 ship window.

D. **Surgical OSM pedestrian exception per S178** — pick specific OSM `osm_id`s that bypass the bad edges (e.g., the actual sidewalk along Salem Green) via the existing allowlist pattern. Stays inside the surgical-only constraint that was just restored. ~1 session.

Recommend confirming the diagnosis first with a quick test before picking a fix: run the Heritage Trail's offending leg as a P2P call (same start/end points as tour leg N→N+1, no GPS) and see if the resulting polyline is the same one that crosses the building in the tour render. If yes → graph-data fix (A or B or D). If no → there IS a hidden code-path divergence I missed, and we'd start over.

### Operator confirms the divergence and proposes address-anchoring

That confirms the actual divergence: the per-leg P2P call `route(POI1.lat, POI1.lng, POI2.lat, POI2.lng)` and the "tap POI2 from near POI1" call `route(GPS_near_POI1, POI2.lat, POI2.lng)` produce different polylines because **`snap(POI1_centroid)` lands on a different graph node than `snap(GPS_on_sidewalk_near_POI1)`** — the POI centroid can be inside or behind a building, the GPS is on the road, so the two snap nodes can be on different sides of the building. Same Dijkstra algorithm, different start node → different shortest path → tour leg's path crosses the building, P2P does not.

Operator's proposal: each tour stop should anchor to a street address (not a centroid lat/lng), so the routing input is a road-side point by definition.
