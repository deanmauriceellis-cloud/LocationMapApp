## Architectural Rule Captured

**POI icons are per-application, not server-side.**

- Servers (cache-proxy / PostgreSQL) return raw POI data only — name, lat/lng, category, type, tags
- Each Android app module supplies its own icon mapping
- Salem app: witch-themed circular icons (gothic purple/black/green)
- Generic LocationMapApp would use clean utility icons; a Vampires app would use different icons again
- The `custom_icon_asset` column on POI rows is the only server-side icon field — and it exists only for paid merchant overrides
- Master plan Step 10.1b (pluggable `MarkerIconProvider` in `:core`) is the formal architectural shape of this rule

## ROOT CAUSE OF SESSION 92 CRASH (the actual one)

User reported "the app was crashing when working on the new icons." Reconstructed from emulator-5570 ANR trace at `/data/anr/anr_2026-04-06-21-52-14-034`.

### The crash
**ANR — Input dispatching timed out (5010ms)** on `com.example.wickedsalemwitchcitytour.ui.SalemMainActivity`. Main thread blocked allocating bitmaps. Earlier ANR on `SplashActivity` at 21:51:29 (5155ms FocusEvent timeout) — same root cause earlier in the launch.

### The stack
```
MarkerIconHelper.labeledDot(MarkerIconHelper.kt:546)
  ↑ Bitmap.createBitmap (4 nested levels — composed bitmap from sub-bitmaps)
SalemMainActivity.narrationIconForZoom(SalemMainActivity.kt:1073)
SalemMainActivity.refreshNarrationIcons$app_salem_debug(SalemMainActivity.kt:1088)
SalemMainActivity$setupZoomSlider$4.onZoom(SalemMainActivity.kt:812)
org.osmdroid.views.MapView.setZoomLevel(MapView.java:521)
org.osmdroid.views.MapController$MapAnimatorListener.onAnimationUpdate(MapController.java:519)
android.animation.ValueAnimator.animateValue → doAnimationFrame
```

### Why
`refreshNarrationIcons` does have a `zoomBucket` guard, but when the user zooms across a bucket boundary (e.g., 17 → 18), ALL **817 narration markers** get their icons recalculated **synchronously on the main thread in one pass**.

At `zoom >= 18`, the call goes to `MarkerIconHelper.labeledDot(this, type, name)`. labeledDot's cache key is `"ldot|$color|$typeLabel|$nameLabel"` — name-specific. Since every narration POI has a unique name, the cache yields **817 misses on the first zoom into bucket 3**.

Each cache miss:
1. Allocates a new `Bitmap.createBitmap`
2. Calls `loadCircleIcon()` which opens an asset, decodes a PNG (800-byte JPEG-like decode), scales with `Bitmap.createScaledBitmap`
3. Draws sub-bitmaps onto the canvas
4. Wraps in `BitmapDrawable`

817 PNG decodes + scales + composites on the main thread = ~10 second main-thread block. Android shows ANR at 5s.

The same pattern likely repeats at the 17 → 18 transition for `dot(this, type, 20)` and earlier transitions, but `dot()` caches by `"cdot|$category|$sizePx"` (~30 unique categories) so misses only ~30 times, not 817.

Secondary symptom: tile cache SQLite connection pool also stalls because the main thread is unresponsive and can't drain message queues, causing osmdroid background tile threads to back up. SELinux denials on FUSE storage are a red herring for a different (already-existing) issue.

### Fix options
1. **Don't use `labeledDot` at zoom 18 for narration markers.** Just use a bigger circle icon — `MarkerIconHelper.dot(this, type, 32)` — same cache hit rate as the smaller dots.
2. **Pre-warm the labeledDot cache off the main thread** at app start (background coroutine builds all 817 entries).
3. **Incremental refresh** — process N markers per frame instead of all at once.
4. **Skip labeledDot entirely** — the witch icons are the brand. Labels were the old approach for the colored-dot world.

Recommended: **Option 1.** The whole point of the new circle icons is they're recognizable without labels. labeledDot was the workaround for tiny colored dots. With proper icons, names should be on tap, not on the marker itself.

### User direction (after report)
User wants to **keep** labeledDot — keep small icons at high zoom and replicate the existing visual. So Option 1 is rejected. The fix needs to make labeledDot performant without changing the look.

### Second critical bug found
`MarkerIconHelper.kt:37` — `MAX_CACHE_SIZE = 500`. The LRU cache cannot hold 817 narration POIs. Even after pre-warming:
- First 500 stored
- Next 317 evict the oldest 317
- Subsequent zoom-bucket transitions miss cache → re-allocate → thrash forever
- Other markers (POI dots, vehicles, GPS, METAR) also get evicted

### Surgical fix plan
1. **Bump `MAX_CACHE_SIZE`** from 500 → 2000 to fit 817 narration + ~30 POI dots × 4 sizes + system markers without thrashing
2. **Filter `refreshNarrationIcons` to viewport** — only iterate markers whose `BoundingBox` contains the position. Off-screen markers don't need re-icon. Reduces ~817 to ~50-100 typically.
3. **Add `"shop_retail" to "shopping/gift_shop"`** to `CIRCLE_ICON_MAP`
4. Build → install on tablet (emulator dead) → user visual verification

This preserves labeledDot, keeps the small icon visual, and eliminates both the ANR and the cache thrashing.

### Applied
- `MarkerIconHelper.kt:37` — `MAX_CACHE_SIZE` 500 → 2000
- `MarkerIconHelper.kt:165` — added `"shop_retail" to "shopping/gift_shop"` (covers 122 businesses)
- `SalemMainActivity.kt:1080-1097` — viewport filter in `refreshNarrationIcons` + debug log of refreshed/total

Build: `:app-salem:assembleDebug` SUCCESSFUL (22s)
Install: tablet HNY0CY0W — Success

**Known limitation of the viewport-only approach:** off-screen markers keep their pre-zoom-transition icon. When user pans to bring them on screen, they show the wrong-size icon for that fraction of a second until the next zoom event. Acceptable for first visual verification — can add a per-marker bucket-tracking + on-scroll refresh if user wants smoother panning.
