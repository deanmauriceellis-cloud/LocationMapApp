Operator first screenshot: NO subtopic header visible. Pulled the device's runtime DB via `adb exec-out run-as ... cat databases/salem_content.db` — confirmed both POIs had the column populated and `PRAGMA user_version = 17`. Decompiled the APK's `poi_detail_sheet.xml` via `aapt2 dump xmltree` — confirmed "Explore subtopics" string was present in the compiled layout. Data was reaching the device, layout was in the APK. Bug had to be in the Kotlin path between Room and the dialog.

Found it in `PoiDetailSheet.kt`:
- Lines 819–839: hand-rolled `salemPoiToJson(poi: SalemPoi): String` serializer that round-trips POI through a JSON string Bundle (because SalemPoi isn't Parcelable). It listed every field the sheet displays — but I never added `narration_subtopics` to it.
- Lines 841–868: matching hand-rolled `parseSalemPoi(s: String): SalemPoi?` deserializer with the same field list.
- Net effect: Room's `narrationSubtopics` was populated correctly, the dialog's `companion.show(poi, ...)` call serialized to JSON dropping the field, the dialog's `onCreateView` deserialized back into a `SalemPoi` with `narrationSubtopics = null`, and `bindSubtopics()` correctly returned early with everything `gone`.

Fix: added `putOpt("narration_subtopics", poi.narrationSubtopics)` to the serializer and the matching `narrationSubtopics = o.optString("narration_subtopics", "").takeIf { it.isNotEmpty() }` to the deserializer.

Rebuild + reinstall; operator confirmed the chip strip + collapsible cards now render correctly on tap of The Burying Point.
