### St. Peter's Church duplicate marker investigation

Operator pressed FAB zoom and saw two St. Peter's Church icons: one flat-on-map at the building anchor, one billboarded floating off-anchor (the upright would snap back to the flat one when the operator nudged the map).

**Initial misdiagnosis** (corrected by operator): I thought the flat label was tile-baked and the upright was the runtime Marker. Operator clarified — both are runtime overlays. So this was a duplicate-rendering bug, not a tile-bake gap.

**Investigation** (`feedback_diagnose_before_changing.md`):
- Verified `tools/tile-bake/style-salem.json` has no worship/cathedral layer — S237 Phase 2 strip is in place. Tile bundle vintage matched the install date.
- Confirmed only one `Marker` per POI in code: `BillboardMarker` added via `SalemMainActivity:2573`, icon bitmap from `MarkerIconHelper.labeledDot` bakes the "category / icon / name" stack into a single bitmap. No FolderOverlay nesting, no second MapView, no parallel POI-painter overlay.
- Investigated S225+ duplicate POI rows (3 St. Peter's-related rows in PG: `st_peters_church`, `st_peters_san_pedro_episcopal_church`, `st_peter_s_church_historic_presentations_and_tours`); ruled out as the cause — operator's symptom was one POI rendered twice, not two adjacent POIs.

**Root cause** (theorized): hardware-accelerated display list staleness. The S237 strip-then-pass-2 design relies on `mv.overlays.removeAll(markers)` taking effect during MapView's `onDraw`. But MapView's RenderNode is not invalidated when only TiltContainer changes — the renderer replays MapView's cached display list (recorded earlier with markers present) → flat ghost markers under the tilt matrix. Pass-2 then layers the upright version on top.

The same caching causes the upright drift after FAB zoom: when osmdroid's `setZoom` updates `mv.projection`, or when the magnify FAB's `mv.animate().scaleX(...)` updates `mv.scaleX`, View framework updates `mv`'s RenderNode property without dirtying TiltContainer's display list. Pass-2 (which lives in TiltContainer's display list) replays with stale `fabScale` + stale projection until something else forces TiltContainer to re-record (e.g. operator's manual nudge fires onScroll → TiltContainer.invalidate via internal listener).

**Option 1 attempt**: `mv.invalidate()` before `super.dispatchDraw(canvas)` in `dispatchDraw`. Built, installed → ANR. `View.invalidate()` is async (schedules next-frame re-record); doesn't help this frame and creates an infinite re-record loop because every dispatchDraw schedules another invalidation. `BufferQueueProducer fps=17 dur=1028ms` confirmed UI thread choke. Reverted.

**Option 2 (shipped)**: per-Marker tilt-suppression flag.
- `BillboardMarker` companion `@Volatile @JvmStatic var tiltActive: Boolean = false` + `override fun draw(canvas, pj)` that early-returns when `tiltActive`.
- `TiltContainer.setTiltDegrees` flips the flag on 0↔>0 transitions and invalidates `mv` once so its display list re-records with no marker draws.
- `TiltContainer.dispatchDraw` no longer strips/restores; pass-2 still iterates `mv.overlays.filterIsInstance<Marker>()` to draw upright billboards.

Built, installed. Operator: "much better."

### Drift + magnify-FAB-doesn't-grow-icons issues (same root cause)

Operator follow-up: upright icons still drift after zoom until manual nudge; magnify FAB (x1→x5) doesn't grow the upright icons.

Diagnosis: same display-list-staleness mechanism as above. `mv.scaleX` and `mv.projection` updates don't force TiltContainer to re-record.

**Fix shipped**:
- `TiltContainer` registers a `MapListener` on the MapView while `tiltDeg > 0`, with `onScroll` and `onZoom` handlers that call `this.invalidate()`. Covers FAB +/- zoom and any pan. Unregistered at tilt=0 so no per-event overhead off-tilt.
- `SalemMainActivity.setupZoomToggle` adds `.setUpdateListener { binding.tiltContainer.invalidate() }` to the magnify FAB's `mv.animate().scaleX(scale).scaleY(scale)` animator — re-records each animation frame so pass-2 sees the in-flight scaleX.

Operator: "much better, no more drift."
