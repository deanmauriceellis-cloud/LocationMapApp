## Pre-work scope

Pre-S214 tour waypoint authoring flow:

- "+ Free waypoint" mode: click anywhere on the map → marker dropped at raw lat/lng.
- Drag-to-move: drop at raw lat/lng.
- Coords persisted to `salem_tour_stops.lat/lng`.
- Router (`salem-router.js`) snaps both endpoints to nearest **graph node** at leg-compute time (S190 `routeBetween`). Mid-block clicks pull all the way to the nearest intersection.
- Failure modes: click-in-river / click-in-parking-lot → router routes to a node 200m away with no operator visibility. "Weird mapping issues" called out by operator.

S214 scope: snap to nearest TigerLine **edge foot** (perpendicular projection) at placement and on every drag. Persist edge binding so future legs / device-side renders can route correctly through mid-block waypoints.

Operator decision (one-question AskUserQuestion): edge-foot snap (chosen), not snap-to-node and not "marker-at-click + tether-line."

## Plan

1. `nearestWalkableEdge(b, lat, lng)` in salem-router.js — grid index over edge polylines, planar SRID-4269 distance.

5. AdminMap UI: free-click + drag-drop use server-returned snapped coords; subtle fuchsia tick on bound edge.

## What shipped

### Router (`cache-proxy/lib/salem-router.js`)

Added in-process spatial grid over walkable edges (re-uses the node grid bounds + cell size — same coordinates), `nearestWalkableEdge(b, lat, lng)` returning `{edge_idx, edge_id, fraction (0..1), snap_lat, snap_lng, snap_m, fullname, mtfcc, from/to_node_idx, edge_length_m}`, and per-edge cumulative segment-length tables so `(segment, t)` projects cleanly to a single fraction along the whole edge polyline. Walks rings outward like `nearestWalkableNode` and stops one ring after first hit. Smoke-tested against four off-network points: Salem Common click → St Peter St 23m, Witch House → unnamed nearby edge 6.6m. Bundle load time unchanged (245ms).

Module return adds `_snapEdge`, plumbed through `server.js` as `deps.salemSnapEdge` for admin-tours.js.

### Schema (`cache-proxy/scripts/2026-05-01-tour-stops-edge-binding.sql`)

`salem_tour_stops` gets two nullable columns: `edge_id BIGINT`, `edge_fraction REAL`. Applied to live PG.

### Admin endpoints (`cache-proxy/lib/admin-tours.js`)

- `GET /admin/salem/snap-edge?lat=&lng=` — preview-only snap for the UI to call on every free-click / drag-end.
- `POST /admin/salem/tours/:tour_id/stops` — when a free waypoint is being created (no `poi_id`), snap server-side, persist `edge_id` + `edge_fraction`, store the snapped lat/lng (not the raw click), echo `snap` block in the response.
- `PATCH /admin/salem/tours/:tour_id/stops/:stop_id` — when lat/lng changes on a free waypoint, re-snap. POI-based stops never snap (POI lat/lng is canonical).
- `GET /admin/salem/tours/:tour_id` — returns `edge_id` + `edge_fraction` per stop.

### Admin UI (`web/src/admin/AdminMap.tsx`, `tourTypes.ts`)

- `TourStop` type gets optional `edge_id`/`edge_fraction`.
- `MoveConfirm` accepts a `snap` prop and renders a fuchsia panel: "Will snap to TigerLine edge: <fullname> · Δ <snap_m>m · edge_id <id>".
- AdminMap fetches `/admin/salem/snap-edge` whenever `pendingStop` changes (drag drop) and feeds the result to MoveConfirm. POI-based stops hide the snap panel.
- After PATCH succeeds, marker is repositioned to the response's `lat`/`lng` (snapped) instead of the operator's drop point — eliminates the visual jitter where the marker would briefly sit at the drop point and then jump to the road on tour refresh.
- TS compile clean (`npx tsc --noEmit` exit 0).

## Add-on: Tour-mode POI preview (mid-session, operator request)

Operator: "I need the ability when I click tours to see what all the POIs are that will show up in tour mode with landmarks and civil etc."

The device's S186 narration gate is global, not tour-specific: during ANY tour, only `is_tour_poi=true` POIs narrate by default; the user can opt in to Hist Landmark + Civic via Layers checkboxes (defaults OFF for tour mode). Operator authoring a tour wants to see what POIs will actually narrate.

### Implementation

- **`PoiTree.tsx`** — `PoiRow` gains `is_civic_poi?: boolean` (already in admin API response; was just missing from the TS type).
- **`AdminLayout.tsx`** — new `tourModeFilter` state `{tourPois: true, histLandmark: false, civic: false}` (defaults match device tour-mode prefs). Plumbed into both `TourTree` (for the controls) and `AdminMap` (for the marker filter).
- **`AdminMap.tsx`** — when `activeTour` is non-null AND `tourModeFilter` is provided, narrow `filteredPois` to the union of:
  - `is_tour_poi=true` (when `tourPois` checked)
  - MassGIS MHC historical (when `histLandmark` checked)
  - `is_civic_poi=true` (when `civic` checked)
  - All-three-off → empty array (operator explicitly hid them all).
- **`TourTree.tsx`** — new `TourModePreview` component renders between Waypoints toolbar and RouteSection. Three checkbox rows with app-wide counts. Bottom row shows the union ("Will narrate" — no double-counting).

Live counts in the current DB: 237 tour POIs, 484 Hist Landmarks, 67 Civic.

### Behavior

When operator picks a tour: map immediately narrows to 237 markers (just `is_tour_poi`). Toggling Hist Landmark adds 484 more (union 707-ish since some overlap). Toggling Civic adds 67. Tour stop markers (numbered draggable waypoints) render on top of this layer regardless — they're not gated by the tour-mode filter.

When operator switches back to the POI view (no `activeTour`), the filter is dormant and the full POI set returns.
