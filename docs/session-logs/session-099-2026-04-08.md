## Phase 9P.4 — Admin POI write endpoints — DONE

### Investigation findings
- Three POI tables, three different schemas:
  - `salem_tour_pois` (curated tour POIs, category column = `category`)
  - `salem_businesses` (restaurants/shops, category column = `business_type`)
  - `salem_narration_points` (Wave 1-4 narration points, category column = `category`, 42 cols incl. merchant fields)
- Only `salem_narration_points` had a `deleted_at` column going in. The other two needed it added.
- `lib/salem.js` did not previously filter `deleted_at` on any read endpoint — soft delete would have been theatrical without a public-read update.
- No prior write endpoints for any of these tables (the `salem` module is read-only). 9P.4 is the first write surface.

### Schema migration (added scope, flagged here for transparency)
- **Added `deleted_at TIMESTAMPTZ` column** to `salem_tour_pois` and `salem_businesses` (already present on `salem_narration_points`)
- **Added partial active indexes** `idx_salem_pois_active` and `idx_salem_biz_active` (`WHERE deleted_at IS NULL`)
- **Updated `cache-proxy/salem-schema.sql`** so the migration is reproducible. Used `ADD COLUMN IF NOT EXISTS` so the file is idempotent on existing deployments.
- **Ran the migration against local PostgreSQL** — verified all three tables now have `deleted_at`.
- **Updated `lib/salem.js` public read endpoints** to filter `deleted_at IS NULL`:
  - `GET /salem/pois` (tour kind)
  - `GET /salem/pois/:id` (tour kind)
  - `GET /salem/businesses`
  - This is a small scope expansion beyond the literal master plan task definition. Reason: without it, soft delete is meaningless because the public app would still show "deleted" POIs. Three-line change, behavior is what users would expect from soft delete.

### `cache-proxy/lib/admin-pois.js` (NEW, ~290 lines)
- Per-kind config: `KINDS = { tour, business, narration }` mapping to table, category column, updatable field whitelist, list ordering.
- **Whitelisted update fields per kind** — no arbitrary column writes. Provenance fields (`data_source`, `confidence`, `verified_date`, `stale_after`) are intentionally writable so the admin tool can mark a POI as freshly verified. Timestamps (`created_at`, `updated_at`, `deleted_at`) are NOT writable — they're managed by the endpoints themselves.
- **JSONB-aware**: `JSONB_FIELDS` set lists every JSONB column across all three tables; `buildUpdateClause` JSON.stringifies those values and casts via `$N::jsonb` in the SET clause.
- **Validation**: `lat ∈ [-90,90]`, `lng ∈ [-180,180]`, `id` non-empty, `kind` in whitelist. Category presence enforced when provided (cannot clear category to empty string). Full taxonomy validation deferred to Phase 9P.6 (TypeScript port of `PoiCategories.kt`) — comment in module.
- **Soft-delete semantics**:
  - PUT and POST .../move REFUSE updates against soft-deleted rows (409 with explanatory error). Admin must restore first.
  - DELETE returns 409 if already deleted, distinguishing from 404 (does not exist).
  - Restore returns 409 if not currently deleted.
  - GET single still returns soft-deleted rows so admin can inspect tombstones before restoring.
  - GET list excludes deleted rows by default, opt-in via `?include_deleted=true`.
- **All write endpoints set `updated_at = NOW()`** automatically.
