### DWELL fix — walk-sim no longer deadlocks on missing TTS engine

Operator: "ship the DWELL fix."

Root cause from S243: walk-sim's TTS-dwell gate at `SalemMainActivity.kt:2162` triggers on `initialPoiSpeaking || initialQueueNonEmpty`. When the TTS engine is missing (Bluestacks ships without Google TTS) or `onInit` returns failure, `NarrationManager` correctly defers enqueues — but the queue stays non-empty and `currentSegment`/state still flips to Speaking from the caller. The dwell loop then waits for `ttsIdle && queueEmpty`, which can't happen without an engine, so the walker hangs until `WALK_SIM_DWELL_MAX_MS = 180_000L` safety cap pops.

Defensive fix: gate the dwell entry on engine availability.

**3 files, surgical:**

- `app-salem/.../tour/NarrationManager.kt` — added public `val isTtsReady: Boolean get() = ttsReady`. The private `ttsReady` was already maintained at every init/shutdown site by S243 Phase 2 instrumentation; just needed a read accessor.
- `app-salem/.../ui/TourViewModel.kt:307` — added `fun isTtsReady(): Boolean = narrationManager.isTtsReady`, matching the existing `isNarrating()` pass-through style.
- `app-salem/.../ui/SalemMainActivity.kt:2166-2178` — read `tourViewModel.isTtsReady()` before the dwell gate. When `!ttsReady && (initialPoiSpeaking || initialQueueNonEmpty)`, emit a `DebugLogger.w("WALK-SIM", "DWELL skipped at step N — TTS engine not ready …; walker would deadlock until safety cap")` and fall through to the next walk step instead of pinning. Existing dwell loop body is unchanged; only the entry predicate is now `ttsReady && (…)`.

The 180s safety cap is preserved as a backstop for any other deadlock mode (e.g. TTS init success but utterance stuck mid-play). This fix is narrow: only handles the "no engine ever ready" case, exactly what S243 traced on Bluestacks.
