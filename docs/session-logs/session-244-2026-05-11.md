### DWELL fix — walk-sim no longer deadlocks on missing TTS engine

Operator: "ship the DWELL fix."

Root cause from S243: walk-sim's TTS-dwell gate at `SalemMainActivity.kt:2162` triggers on `initialPoiSpeaking || initialQueueNonEmpty`. When the TTS engine is missing (Bluestacks ships without Google TTS) or `onInit` returns failure, `NarrationManager` correctly defers enqueues — but the queue stays non-empty and `currentSegment`/state still flips to Speaking from the caller. The dwell loop then waits for `ttsIdle && queueEmpty`, which can't happen without an engine, so the walker hangs until `WALK_SIM_DWELL_MAX_MS = 180_000L` safety cap pops.

Defensive fix: gate the dwell entry on engine availability.

**3 files, surgical:**

- `app-salem/.../tour/NarrationManager.kt` — added public `val isTtsReady: Boolean get() = ttsReady`. The private `ttsReady` was already maintained at every init/shutdown site by S243 Phase 2 instrumentation; just needed a read accessor.
- `app-salem/.../ui/TourViewModel.kt:307` — added `fun isTtsReady(): Boolean = narrationManager.isTtsReady`, matching the existing `isNarrating()` pass-through style.
- `app-salem/.../ui/SalemMainActivity.kt:2166-2178` — read `tourViewModel.isTtsReady()` before the dwell gate. When `!ttsReady && (initialPoiSpeaking || initialQueueNonEmpty)`, emit a `DebugLogger.w("WALK-SIM", "DWELL skipped at step N — TTS engine not ready …; walker would deadlock until safety cap")` and fall through to the next walk step instead of pinning. Existing dwell loop body is unchanged; only the entry predicate is now `ttsReady && (…)`.

The 180s safety cap is preserved as a backstop for any other deadlock mode (e.g. TTS init success but utterance stuck mid-play). This fix is narrow: only handles the "no engine ever ready" case, exactly what S243 traced on Bluestacks.

### `no_overwrite` authoring lock — end-to-end

**PG (idempotent bootstrap):** `ALTER TABLE salem_pois ADD COLUMN IF NOT EXISTS no_overwrite BOOLEAN NOT NULL DEFAULT FALSE` at the top of `module.exports = function(app, deps)` in `cache-proxy/lib/admin-pois.js`, mirroring the S187 `salem_lint_suppressions` pattern. Re-deploy safe; existing 2,035 rows pick up `false`.

**Protected set** (new exported constant `NO_OVERWRITE_PROTECTED_FIELDS`):
`short_narration`, `long_narration`, `historical_narration`, `narration_subtopics`, `description`, `short_description`, `custom_description`, `origin_story`, `year_established`. The flag itself is not in the protected set — operators can always toggle it off.

**Enforcement helper:** new `lockedFieldsThatWouldChange(existingRow, body)` in admin-pois.js — returns the list of protected fields where the incoming body value differs from the stored value (JSONB columns compared by `JSON.stringify`).

**`PUT /admin/salem/pois/:id`:** existing query that fetched only `id, deleted_at` widened to `SELECT *` so the helper has the stored values to compare against. New gate before `UPDATE`: if `req.query.force` is not `'true'`/`'1'` and `lockedFieldsThatWouldChange()` returns a non-empty list, return `409 { error, code: 'NO_OVERWRITE_LOCKED', locked_fields }`.

**`POST /admin/salem/pois/apply-mass-edit`:** new pre-flight in `admin-mass-edit.js` that queries `SELECT id, name, no_overwrite, <protected fields> FROM salem_pois WHERE id = ANY($1) AND no_overwrite = TRUE` against the grouped approvals BEFORE opening the transaction; collects all blocked POIs in one pass and refuses the entire batch with `409 { blocked: [{ poi_id, poi_name, locked_fields }] }`. Single locked row blocks the whole apply unless `?force=true` — by design (the operator's narrative was "any edits to a flagged POI need to be done by hand or flagged as NoOverwrite," so half-applying a batch around a locked row would be worse than refusing).

**Frontend:**

- `web/src/admin/poiAdminFields.ts` — added `no_overwrite` to `UPDATABLE_FIELDS` and `BOOLEAN_FIELDS` so the existing buildDefaults/buildPayload paths handle it as a normal boolean.
- `web/src/admin/PoiTree.tsx` — added `no_overwrite?: boolean` to the `PoiRow` interface (the existing GET returns `SELECT *` so the field flows through automatically; just type-narrowed for callers); added an inline lock indicator (🔒, amber) next to leaf rows where `data.poi?.no_overwrite` is true, with `title="Admin Authored — Do Not Overwrite (NoOverwrite)"`.
- `web/src/admin/PoiEditDialog.tsx` — added an amber "Authoring Lock" panel pinned to the top of the Flags tab containing a single toggle "Admin Authored — Do Not Overwrite (NoOverwrite)" with explanatory hint text. Rewired `onSubmit` to fetch via a closure `doPut(force)`; on `res.status === 409` with `body.code === 'NO_OVERWRITE_LOCKED'`, surface a confirm dialog listing the locked fields and retry with `?force=true` only if the operator accepts.
