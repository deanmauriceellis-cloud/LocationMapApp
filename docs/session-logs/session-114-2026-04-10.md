### Architecture conversation: seven parallel POI taxonomies

Mapped the seven taxonomies that currently exist:
1. `PoiCategories.kt` (intended canonical, 22 categories × 176 subtypes, line-aligned with TS mirror)
2. `web/src/config/poiCategories.ts` (TS mirror, 657 lines)
3. PG `salem_narration_points` — has `category TEXT NOT NULL` + `subcategory TEXT` (but `subcategory` is NULL on all 814 rows; `category` holds 29 legacy lowercase values)
4. PG `salem_tour_pois` — has `category TEXT NOT NULL` + `subcategories JSONB` — and critically, `category` holds **tour-chapter themes** (witch_trials, maritime, literary, landmark, museum, park, visitor_services), a DIFFERENT concept from the POI layer taxonomy
5. PG `salem_businesses` — uses `business_type` + `cuisine_type` + `tags`, no `category` column
6. Bundled SQLite in the APK — single `type` column, old shape, what Android Room entities read today
7. `tools/generate-poi-icons.py` POI_ICONS — fictional 185 (category, slug) pairs

Explained the historical accretion: Phase 9P.A (S98-S101) migrated storage to PG as source of record with `category`/`subcategory` columns designed to match PoiCategories.kt, but the publish loop (Phase 9P.C) was never built, so the app still reads the old shape from the frozen bundled SQLite. The admin tool writes the new shape into PG, but nothing copies that into the APK.

Answered operator's "single template?" question: yes, the common spine is already defined (`PoiCategories.kt`), but it isn't enforced as a DB constraint across the three PG tables, so `salem_businesses` drifted to `business_type`/`cuisine_type` and nothing stopped it.

1. Make `PoiCategories.kt` the enforced vocabulary across PG tables (lookup tables + FK).
2. Backfill migration + "needs subcategory" admin worklist.
3. Phase 9P.C publish loop + Room schema bump + real migration (not destructive fallback).
