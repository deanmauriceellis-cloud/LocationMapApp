### GPS interval thrasher — FIXED
- Root cause: `SalemMainActivity`'s 3-state interval picker (10s / 2.5s / 30s) transitioned on every fix with no hysteresis; each transition called `locationJob.cancel()` → `awaitClose` → `removeLocationUpdates` → re-register, which tore down the FLP subscription for a couple of seconds. With speed oscillating around the 20 mph boundary during a car tour, this fired 19 times in 15 min.
- **Two fixes applied together:**
  1. `core/.../LocationManager.kt` — added `updateRequestParams(intervalMs, minIntervalMs)` which calls `fusedClient.requestLocationUpdates()` again with the existing callback. GMS merges onto the live subscription; flow stays open. Also added `activeCallback`/`activeIntervalMs` bookkeeping with clean-up in `awaitClose`.
  2. `app-salem/.../MainViewModel.kt` `restartLocationUpdates()` — now tries the in-place path first; only falls back to `getLocationUpdates` + relaunch if no live subscription.
  3. `app-salem/.../SalemMainActivity.kt` — interval picker reduced from 3 states to 2 (2.5s moving/narrating, 30s idle). The >20 mph → 10s branch was the biggest thrasher and the battery saving is irrelevant during a 1-hr tour. Added `GPS_INTERVAL_DWELL_MS=10_000` dwell debounce: a candidate interval must persist for 10s before being applied. Brief stops at traffic lights won't flip to the idle branch.
- Compile + install: `ANDROID_SERIAL=HNY0CY0W ./gradlew :app-salem:installDebug` → BUILD SUCCESSFUL (52s).
