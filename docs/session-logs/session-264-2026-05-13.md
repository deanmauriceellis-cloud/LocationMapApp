### Root causes identified (4)

1. **60-second default rate** ‚Äî `MainViewModel.kt:140` hardcoded `currentIntervalMs = 60_000L` from the S126 battery-conservative era. Operator's high-zoom tracking use case needed 1 Hz.
2. **Walk-sim race kills tour-mode flow** ‚Äî `setManualLocation()` called `stopLocationPolling()` on every walk-sim tick (~1 Hz). When the tour engine called `restartLocationUpdates(2500ms)`, walk-sim's next tick fired `stopLocationPolling()` ~14 ms later, cancelling the fresh fast Flow. Operator's mis-tap-of-Walk-button (which they corrected immediately) was enough to trigger this and leave the app stranded.
3. **Zombie callback leak in `LocationManager.kt`** ‚Äî `awaitClose { fusedClient.removeLocationUpdates(callback) }` could fire BEFORE the `requestLocationUpdates` Task finished registering. GMS order: remove (no-op, callback not yet registered) ‚Üí register (succeeds, no remove pending) ‚Üí callback alive forever with nothing collecting it. Trip log evidence: at 19:30, two concurrent `onLocationResult #N` counter series running ‚Äî counter #234+ (zombie) firing every 3 s but never reaching ViewModel, counter #57+ (live flow) reaching ViewModel only every ~15 s, producing the "fix every 15 seconds" sluggish feel.
4. **`stopWalkSim()` doesn't restore `_locationMode = GPS`** ‚Äî after walk-sim stops, the cascade leaves the app in MANUAL mode with the GMS subscription dead. Map stays frozen until a manual GPS-toggle FAB tap or app restart.

### Patch shipped (~40 LOC, 3 files, no new deps)

| File:line | Change |
|---|---|
| `MainViewModel.kt:130, 140-142` | Default interval `60_000L / 30_000L` ‚Üí `1_000L / 500L` |
| `MainViewModel.kt:134-138` | `onPermissionGranted()` early-return now also flips `_locationMode = GPS` (in case caller comes from MANUAL with subscription still alive) |
| `MainViewModel.kt:208` | Removed `stopLocationPolling()` from `setManualLocation()` ‚Äî gating now relies on `_locationMode` check that already exists in the collector |
| `MainViewModel.kt:212-220` | New `endManualMode()` ‚Äî explicit MANUAL‚ÜíGPS flip without resubscribing |
| `SalemMainActivity.kt:2249, 2275, 2288` | All 3 `stopWalkSim*` variants now call `viewModel.endManualMode()` so the map recovers on the next real fix |
| `LocationManager.kt:140-155` | `registerCompleted` / `cancelArrivedBeforeRegister` flags ‚Äî `addOnSuccessListener` cleans up a zombie if cancel arrived before register completed |

Both flags are local `var`s ‚Äî they live in the callbackFlow closure and are touched only on the Main looper (`addOnSuccessListener` default + `awaitClose` on the collecting coroutine), so no volatile needed. The first build attempt failed with `@Volatile`-on-local-var (not applicable to that target); dropped the annotation and the second build went green.
