## P3b done — live re-routing

### `DirectionsSession` (`app-salem/.../routing/DirectionsSession.kt`)

- Immutable per-route state. Constructor takes `destination: GeoPoint` + `polyline: List<GeoPoint>`. Caller creates a new instance on every reroute.
- `onLocation(point) → SessionAction`:
  - `Arrived` when within 15 m of destination (haversine). Checked first so the natural last-segment shortcut from sidewalk → driveway → door does not produce a reroute loop.
  - `Reroute(from, driftM)` when cross-track drift >25 m for **2 consecutive fixes**. Counter resets when a fix lands within 25 m again.
  - `OnPath` otherwise.
- Geometry helpers: `distanceToPolyline`, `pointToSegmentM` (local equirectangular projection around segment midpoint — sub-0.1% error for sub-km segments at Salem latitude), `haversineM`. All `internal` so a P2c JVM unit test can hit them directly.

### `TourViewModel` wiring

- New `private var directionsSession: DirectionsSession?` and `private var isRerouting: Boolean` flag.
- `getDirectionsTo(dest)` and reroutes go through one private suspend `computeAndApplyRoute(from, dest)` — emits the route on `_walkingRoute`, replaces the session.
- `onLocationUpdate(point)` now also calls `handleActiveDirectionsLocation(point)` after the existing `tourEngine.onLocationUpdate(point)`.
- `handleActiveDirectionsLocation`:
  - No-op when `directionsSession` is null (multi-stop tour previews leave the session null deliberately) or `isRerouting` is true (mute drift while a reroute is in flight).
  - On `Arrived` — calls `clearDirections()` (route disappears, session cleared).
  - On `Reroute` — sets `isRerouting=true`, kicks off `computeAndApplyRoute(action.from, session.destination)`, clears the flag in `finally`.
- `clearDirections()` now also nulls the session and `isRerouting`.

## P3c done — out-of-Salem simulated walk from MBTA

### Constants + walkTo branch (`SalemMainActivityDirections.kt`)
- `MBTA_SALEM_STATION = GeoPoint(42.524, -70.8989)` — Bridge St / Riley Plaza area, the canonical "you just stepped off the train" origin (>50% of Salem visitors arrive by rail).
- `OUT_OF_SALEM_KM = 3.0` — threshold; beyond this, walkTo() forks to the simulator path.
- `walkTo(destination)`: when `distanceBetween(lastGpsPoint, MBTA) > 3 km`, calls `startSimulatedWalkFromMbta(destination)` instead of the normal `tourViewModel.getDirectionsTo(...)`.

### `startSimulatedWalkFromMbta` + walk loop
- Computes route via `tourViewModel.computeWalkRoute(MBTA, destination)`.
- Publishes the polyline on `_walkingRoute` and arms a P3b `DirectionsSession` via `tourViewModel.publishWalkRoute(destination, polyline)` — so arrival within 15 m of destination clears the route exactly the way it does for real-GPS routes.
- Drives the simulator via new `walkSimAlongPolyline(polyline, label)` helper. That helper:
  - Reuses existing `walkMutex`, `walkSimJob`, `walkSimRunning`, FAB UI hooks, `acquireWalkWakeLock` / `releaseWalkWakeLock`, and the now-`internal` `interpolateWalkRoute` interpolator.
  - Does **not** touch tour state or `narrationGeofenceManager.walkSimMode` — this is a "get me to the POI" walk, not a tour walk.
  - Cancels any in-flight tour walk (via `stopWalkSimExternalAndJoin`) and any HTTP walk (via `DebugEndpoints.cancelWalkAndJoin`) before starting.

### `TourViewModel` additions
- `computeWalkRoute(from, to): ArrayList<GeoPoint>?` — pure compute, no side effects.
- `publishWalkRoute(destination, polyline)` — display + arm P3b session for an arbitrary polyline. Uses an internal `polylineLengthM` haversine summer to populate the WalkingRoute distance/duration metadata.
