## Narration Not Triggering on Real GPS — Bug Fix

**User report:** Walking in Salem, near POIs (restaurants etc.) — no narration triggered.

**Root cause analysis — 2 bugs found:**

### Bug 1: 100m dead zone blocks narration system
- **File:** `SalemMainActivity.kt:1217`
- GPS observer has a 100m dead zone: if movement < 100m from last processed point, entire update is `return@observe`
- This skips `updateTourLocation()` which feeds `NarrationGeofenceManager.checkPosition()`
- Geofence entry radii are 20-50m — user walks right through them between 100m updates
- **Fix:** Added `updateTourLocation(point)` call inside the dead zone block. Map/marker still skip jitter, but narration system gets every GPS update.

### Bug 2: 60-second GPS interval at walking speed
- **File:** `SalemMainActivity.kt:1204`
- Adaptive interval: >20mph → 10s, else → 60s
- At walking speed (~3mph, 1.5 m/s), GPS updates only every 60s = ~90m between checks
- Combined with 100m dead zone, narration system gets updates every 100-200m
- **Fix:** Added third tier: when `narrationGeofenceManager` is active, use 5s interval (min 2s). At walking speed: ~7.5m between updates, well within geofence range.

## Narration Queue Race Condition — Bug Fix

**User report:** Walking silently for minutes, no narration despite POIs nearby.

**Debug logging added:** NARR-QUEUE, NARR-PLAY, NARR-STATE, NarrationMgr tags throughout pipeline.

**Logcat evidence:**
- TTS played "Vampire Ghost Adventures" successfully at 04:38:26
- TTS finished → Idle at 04:38:45, queue was empty (0)
- 0.6s later Artemisia Botanicals queued, then 16 more POIs piled up (queue grew to 21)
- `currentNarration` stuck on "Vampire Ghost Adventures" — never cleared
- No TTS output after that — queue filled but never drained

**Root cause:** Race condition in auto-advance observer. When TTS goes Idle with empty queue, code waits 5s for reach-out. During that delay, new POIs get queued via geofence. After delay, code checks `if (narrationQueue.isEmpty())` — it's NOT empty, but there was no else branch to play the queued items. Code fell through. And `currentNarration` never cleared, so all future entries went to queue-only branch.

**Fix:**
1. Clear `currentNarration = null` when narrationState goes Idle
2. After silence delay, if queue filled up during wait, call `playNextNarration()`

**Files modified:**
- `SalemMainActivityNarration.kt` — race condition fix + verbose debug logging
