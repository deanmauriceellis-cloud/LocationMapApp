### Work block 2 — CLEARTEXT block fix ("strange behavior")

Operator: "look at the logs and add more verbose logging, strange behavior". Pulled live Lenovo logcat by PID. Found smoking gun:

```
05-12 00:03:05.156 E/WeatherVM: fetchWeatherDirectly FAILED:
  CLEARTEXT communication to 10.0.0.229 not permitted by network security policy
  UnknownServiceException at RealConnection.connect
```

Android API 28+ blocks plain HTTP by default. S249's INTERNET grant in src/debug/AndroidManifest.xml wasn't enough — OkHttp threw UnknownServiceException before ever hitting the wire. Every parked-service network call would silently fail the same way. (The 900+ MBTA bus markers visible in the S243 MapDump were stale-cached from when the cache-proxy was reachable some other way.)

Created new `app-salem/src/debug/res/xml/network_security_config.xml` allow-listing cleartext to 10.0.0.229 + localhost + 127.0.0.1 (debug-only — release builds do not include this overlay). Referenced from `app-salem/src/debug/AndroidManifest.xml` via `<application android:networkSecurityConfig="@xml/network_security_config" tools:replace>`. Build + reinstall.

S243 instrumentation in WeatherViewModel was already solid — the existing `DebugLogger.i/e` calls surfaced the failure. No additional logging needed; the problem was a missing config, not missing visibility. (My initial logcat tag filter missed it because the actual TAG is "WeatherVM" not "WeatherViewModel" — operator-facing finding for future filter use.)
