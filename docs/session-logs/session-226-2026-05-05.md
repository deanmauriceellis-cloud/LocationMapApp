### Post-deploy fix — haunt POIs were invisible on the map

After the first deploy the operator placed a second admin-created POI ("Beverly/Salem City Line", category=CIVIC, skeleton haunt with blank tuning columns), re-baked, redeployed — and the POI never rendered on Lenovo even though `Show all POIs → true`.

Root cause: `SalemPoiDao.findNarrated()` (the marker loader's underlying query) requires `is_narrated = 1`. The S225 admin-create flow doesn't flip `is_narrated`, and the operator hadn't authored narration for the City Line. So the POI sat in PG with `is_narrated=false` and was filtered out before the layer-visibility check ever ran. Confirmed via `Loading 1303 narration markers ... total=1303` (City Line not in either count).

Fix (single-query change):

```kotlin
// SalemPoiDao.findNarrated() — accept haunt-only POIs as map markers
@Query("""
  SELECT * FROM salem_pois
  WHERE (is_narrated = 1 OR (haunt_sprite_id IS NOT NULL AND haunt_enabled = 1))
    AND (default_visible = 1 OR is_tour_poi = 1 OR is_historical_property = 1 OR is_civic_poi = 1)
  ORDER BY priority ASC, name ASC
""")
```

Build + reinstall + relaunch logcat: `Loading 564 narration markers (showAll=false, total=1304 ...)` — the +1 is the City Line POI. End-to-end working: an admin-created POI with only a haunt configured now appears on the explore map regardless of narration. Geofence narration system unaffected because its downstream filters (e.g. `has_announce_narration`) still short-circuit non-narrated POIs.

Side benefits of this change discovered during the diagnosis (logged here so future sessions don't re-trip them):

- The S225 PoiCreateDialog default-stamps `data_source='admin_create'`, `admin_dirty=TRUE`, `is_civic_poi` derived from category. It does NOT default `is_narrated=true`. That's correct — admin-create rows shouldn't lie about having narration content. The marker loader is what needed loosening, not the create defaults.
- `cache-proxy/lib/admin-categories.js:131` deliberately stamps `default_enabled=FALSE` on every operator-created category, so any POI placed in a brand-new category will be invisible until either (a) the operator flips the corresponding Layers pref or (b) the POI carries an override flag (is_civic_poi / is_tour_poi / is_historical_property). Confirmed no new categories were actually persisted today — operator's "+ Add new category" attempt produced no new rows in `salem_poi_categories`. Carry-forward: future sessions should expect operator-created categories to need the operator to flip the Layers pref before POIs in them render.
