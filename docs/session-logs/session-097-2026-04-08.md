User clarified:

- Historical narrative is a separate stream from `~/Development/Salem`
- Two viewing modes needed: free-roam (current ambient) vs historic tour (curated, ambient-suppressed)

Spawned an Explore agent for a targeted survey. Findings:
1. Historical content tables are populated (49 figures, 500 facts, 40 events, 200 sources) but ALL FK columns to POIs are NULL — the bridge has never been built
2. Salem JSON layout: 38,788 files across 11 subdirectories. Largest sets: NPC responses, relationships, NPCs, primary sources
3. Tour mode partially exists in code: `TourEngine.kt:72` has `var ambientModeEnabled: Boolean = false`, line 314 checks tour state for ambient suppression — but it's implicit, not a formal mode toggle
4. Per-category visibility is monolithic: single global on/off per category via SharedPreferences `prefKey` (e.g., `poi_food_drink_on`). No per-mode support today
5. POI ↔ historical FK linkages: confirmed all NULL

User interrupted my response to clarify: they had been finishing other content work (Salem session 044 — 202 daily Oracle newspapers, ~13.5 hours TTS-ready, with `events_referenced` JSONB field). User confirmed:
- Salem JSON entities use `building_id`, never POI ID — by design
- Translation layer must live on LocationMapApp side (`salem-content/`)
- Newspapers are date-keyed, join to POIs via events_referenced → events → buildings → POIs
- They never opened the bundled SQL this session, so survey results are still accurate

Drafted three-phase architecture in response. User approved with "I am good with your recommendations, update the master plan and session end, start kick this off next session."
