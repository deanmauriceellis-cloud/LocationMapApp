### Implementation

Six surfaces. Followed `feedback_leverage_existing_assets.md` — extended the existing toolbar / `KatrinaCameraManager` / `BuildDefaults` patterns rather than building parallel infrastructure.

1. **`BuildDefaults.kt`** — added `const val GPS_BURST_ENABLED: Boolean = BuildConfig.RECON_DEFAULTS`. Same `RECON_DEFAULTS` BuildConfig field already used for the seven existing recon-vs-retail flips, so no new build.gradle wiring.

2. **`toolbar_two_row.xml`** — added `toolbarGpsBurstIcon` ImageView immediately after `toolbarCameraIcon` in the icon row. Default `visibility="gone"` (made visible by AppBarMenuManager only when `BuildDefaults.GPS_BURST_ENABLED`). Reuses `ic_camera_recon` drawable; tint distinguishes from the existing button when ON (white→red).

3. **`MenuEventListener.kt`** (core) — added `fun onGpsBurstToggled(enabled: Boolean)` with stub default routing through `onStubAction("gps_burst:$enabled")` so non-Salem hosts compile without overriding.

4. **`AppBarMenuManager.kt`** — extended `setupSlimToolbar(...)` with `gpsBurstIcon: ImageView? = null` parameter. Wires session-scoped toggle: tap flips `gpsBurstActive`, repaints tint (WHITE↔#E53935 red), fires `menuEventListener.onGpsBurstToggled(newState)`. State is **NOT persisted** — every cold start opens with burst OFF so the operator can't accidentally leave it on overnight and burn the gallery. The whole `gpsBurstIcon?.let { ... }` block short-circuits to `View.GONE` when `GPS_BURST_ENABLED` is false; R8 collapses the dead branch.

5. **`GpsBurstCameraManager.kt`** (NEW, ~370 LOC at `app-salem/.../userdata/`) — headless CameraX controller. Highlights:
   - **CameraX `ImageCapture` use case bound to activity lifecycle, NO Preview surface** — silent background capture, doesn't steal map focus, auto-pauses with the activity. `ProcessCameraProvider.bindToLifecycle(activity, DEFAULT_BACK_CAMERA, imageCapture)`. CameraX is already a dependency (used by ReconCaptureActivity).
   - **Self-owned 1 Hz `LocationManager` flow subscription** via `lifecycleScope.launch { locationManager.getLocationUpdates(1000L, 1000L).collect { ... } }`. Distinct from MainViewModel's subscription so EXIF gets raw, unclamped GPS (bypasses the Samantha-statue clamp). GMS coalesces multiple FusedLocationProvider subscriptions, so the existing observer is unaffected.
   - **3-second throttle** by tracking `lastShotAtMs`. If `now - lastShotAtMs < 3000ms`, skip. So a 1 Hz fix stream produces ~1 photo per 3s while the toggle is ON.
   - **EXIF stamp:** lat/lon/alt/speed/track + accuracy + GPS_IMG_DIRECTION (compass heading via `TYPE_ROTATION_VECTOR` listener, mirrors KatrinaCameraManager's pattern) + DateTime + Make/Model/Software + UserComment with all numeric fields.
   - **Filename:** `burst_YYYYMMDD-HHMMSS_<lat>_<lon>.jpg` so the gallery thumbnail strip already shows time + position. Negative coordinates have `-` swapped to `n` to keep filenames clean across file managers.
   - **MediaStore publish** to `Pictures/WickedSalemRecon/` (matches operator's chosen folder name; distinct from existing `Pictures/WickedSalem-Recon` recon album).
   - **Permission flow** mirrors KatrinaCameraManager: registers an `ActivityResultLauncher<String>` for `Manifest.permission.CAMERA`. If not granted at toggle-ON, launches the prompt and resumes start on grant. Refuses to start without `ACCESS_FINE_LOCATION`.
   - **Lifecycle:** `start()` binds camera + starts location job, `stop()` cancels job + `cameraProvider.unbindAll()`. `onResume()/onPause()` register/unregister the rotation-vector sensor for compass heading. Toast on first photo + on toggle-OFF (with shot count).
   - **Why duplicated EXIF/MediaStore code instead of refactoring `KatrinaCameraManager`:** keeping stable code untouched — refactor would have required retesting the existing recon-camera path. Duplication is ~80 LOC; can refactor into a shared `ReconPhotoStore` later if it bothers us.

6. **`SalemMainActivity.kt`** — six edits: import the new manager class, `@Inject internal lateinit var burstLocationManager: LocationManager`, lazy `gpsBurstCameraManager` instance, register `gpsBurstPermissionLauncher`, pass `toolbarGpsBurstIcon` to `setupSlimToolbar`, call `gpsBurstCameraManager.registerPermissionLauncher(...)` in `onCreate`, hook `onResume()/onPause()` for sensor lifecycle, override `onGpsBurstToggled(enabled)` in `menuEventListenerImpl` to call `setEnabled()`. Every call site is gated by `if (BuildDefaults.GPS_BURST_ENABLED)` so R8 dead-codes the manager + sensor + camera bind in retail builds.

### Round 2: throttle + poll-rate tuning + flash-off

Operator: "seems we are slower than 3s a picture". Histogram of the 484 inter-shot gaps: **min 3 s, median 4 s, mean 4.26 s, p95 10 s, max 84 s**. Bimodal shape (194 @ 3 s + 263 @ 4 s) — diagnosed as the on-device FusedLocationProvider serving fixes ~every 2 s, so the throttle floor (3000 ms) was clearing on the second-fix-after-last-shot most of the time, not the first. 24 outliers in the 11–30 s bucket and one 84 s gap are real GPS dropouts (signal shadow / between-buildings).

Three changes in `GpsBurstCameraManager.kt`:

1. **GPS poll 1 Hz → 2 Hz** — `getLocationUpdates(intervalMs = 500L, minIntervalMs = 500L)`. Finer-grained throttle moments.
2. **Throttle 3 s → 2 s** — `THROTTLE_MS = 2_000L`. Toast string uses `THROTTLE_MS / 1000` so it auto-updates to "every 2s".
3. **Flash forced OFF, belt + suspenders** — `.setFlashMode(ImageCapture.FLASH_MODE_OFF)` on the builder *and* `capture.flashMode = FLASH_MODE_OFF` on the live instance.
