### Headline (S251 close)

Operator reported "I don't see MBTA trains/buses/stations" and "weather hangs / failed to load weather data" — despite S250 declaring MBTA end-to-end. Two genuinely separate root causes diagnosed, both fixed:

1. **Code race: one-shot `fetchMbtaStations()` ran in `SalemMainActivity.onStart()` ~12 s BEFORE the operator toggled SuperAdmin on, silently bailed on the V1_OFFLINE_ONLY guard, and never retried.** Refresh-loop fetches (trains/subway/buses) self-recovered on their 30 s ticks; station + bus-stop one-shots stayed empty for the rest of the session. Fix: added `kickDeferredSuperAdminFetches()` hook that re-fires the one-shots when the operator flips SuperAdmin → ON. Log line `MBTA stations success — 262 stations` at 01:24:19.431 confirmed working post-fix.

2. **Infrastructure: `ufw` was silently blocking inbound `tcp/4300` on this dev box's wired interface `enp2s0` (where the cache-proxy actually listens at 10.0.0.229).** ICMP flowed (so ping succeeded). MBTA polls *appeared* fine because OkHttp's connection pool kept one warm socket alive across 30 s ticks — never opened a new connection. Cold sockets from Weather / METAR / Webcams / FlightHistory always failed at the 15 s connect timeout. Operator ran `sudo ufw allow 4300/tcp`; TCP probe from Lenovo went from 5/5 timeouts → 20-140 ms in three tries. Weather verified working post-fix.

Bonus: replaced 17 silent `if (FeatureFlags.V1_OFFLINE_ONLY && !SuperAdminMode.allowNetwork) return` short-circuits across TransitVM / WeatherVM / AircraftVM with logged `SuperAdminMode.networkBlocked(TAG, action)` calls. Future invisible-bail bugs of the same shape will now surface in logcat with the function name, V1 flag state, available + enabledFlag.
