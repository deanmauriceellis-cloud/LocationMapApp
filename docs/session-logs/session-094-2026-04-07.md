## CORRECTION — Categorical Distinction (post-commit `5f17650`)

User correction: "locationmapapp DOES NOT track anything except by hash. RadioLogger, WWWatcher are proactive tools for threat management."

The Session 94 synthesis above (and the `5f17650` commit) framed this as "three collectors" on equal footing in a shared scanning architecture. **That framing is wrong.** The correct framing is two categories, not three peers:

### Two categories

**Proactive threat management tools (operator-owned, exempt):**
- **WWWatcher** — designed for discovery and tracking. Holds raw data. Runs analyzers. Escalates. Phase 20 active multispectrum sensor on operator premises.
- **RadioLogger** — designed for continuous radio environment logging on operator test fleet. Holds raw CSV data. Runs RadioLogger-class analyzers (the same five that WWWatcher session 068 ported into Rust).

Both tools have a threat management *purpose*. Both hold raw values. Both run the analyzer DNA. Both are operator-owned and exempt per OMEN-008.

**Hashed-observation contributor (consumer hardware, constrained):**
- **LocationMapApp Salem** — a tourist app. Its RadioIntelligence contribution is to observe ambient public broadcasts, **hash on device**, and forward hashes to a relay. Never holds raw values. Never runs analyzers. Never tracks. Never escalates. The hash→raw lookup table lives only on the operator's separate home box, gated by criminal-exception escalation. **None of that machinery exists in LocationMapApp's codebase or on the visitor's phone.**

LocationMapApp Salem is not a "scanning peer" of WWWatcher/RadioLogger. Calling it one mischaracterizes the architecture and would propagate incorrect language into the Privacy Policy, code design, OMEN reports, and downstream sessions.

### What this means concretely

- **Salem app code path (when post-Privacy-Policy approval permits it):** observe ambient broadcasts → hash on device with shared salt → ship hashes to relay → forget. No analyzers. No raw retention. No threat awareness. No alerting. No lookup table. The Salem app doesn't even know what device a hash represents.
- **The unified hash space (OMEN-008 §5) is a property of the home box, not of LocationMapApp.** The cross-reference happens at the home box because it has both the WWWatcher raw store and the incoming Salem hashes. LocationMapApp does not perform the matching; it just contributes hashes.
- **The shared analyzer DNA (WWWatcher session 068's `crates/wwwatcher-anomaly/src/rf_*.rs`) lives in WWWatcher and RadioLogger.** None of it ever runs in LocationMapApp Salem.
- **The Privacy Policy's load-bearing first-paragraph claim** must be: "LocationMapApp never sees or stores device identifiers — only one-way cryptographic hashes computed on your phone before any data leaves your device." Everything downstream of that claim must remain consistent.
