## P4 done — Web router in cache-proxy

### `cache-proxy/lib/salem-router.js` (new)

JS port of `core/.../routing/RoutingBundle.kt` + `Router.kt`. Reads the same `app-salem/.../assets/routing/salem-routing-graph.sqlite` the APK ships, so web and Android produce identical routes for identical inputs.

- **Bundle loader** — opens with `better-sqlite3({readonly: true})`, validates `meta.schema_version=1`, reads `nodes` (ORDER BY id) into `Float64Array`s + a `Map` for id→idx, reads `edges` (id,source,target,length_m,mtfcc,fullname,geom_polyline). Mirrors `JdbcRoutingBundleLoader.kt` field-for-field.
- **CSR build** — degree pass + prefix-sum offsets + insertion cursor. Each edge contributes to BOTH endpoints (undirected), with a `reversed` flag so the polyline can be emitted in travel direction.
- **Spatial grid** — 0.0015°×0.0020° cells (~165 m), expanding-ring KNN over walkable nodes only, planar SRID-4269 degree distance (matches `tiger.<->`).
- **Dijkstra** — binary min-heap with parallel `Float64Array` distances + `Int32Array` indices, lazy delete via `dist[u]` check, identical predecessor reconstruction.

### Endpoints

- `GET /salem/route?from_lat&from_lng&to_lat&to_lng[&source=live|bundle]` — defaults to bundle.
- `POST /salem/route-multi` — `{stops: [{lat,lng},…], source?}`, pairwise-routed concatenation.
- `GET /salem/route/meta` — bundle metadata for clients.

Web side hits `/api/salem/...` and Vite/cache-proxy strip `/api` → server endpoints live at `/salem/...` to match the existing `lib/salem.js` pattern.

### Live fallthrough (`?source=live`)

- Lazy-init `pg.Pool` against the `tiger` DB. **Peer auth requires the unix socket**, so the pool is built with `host: '/var/run/postgresql', database: 'tiger'` — without `host`, node-pg defaults to TCP and SCRAM-asks for a password we don't have. (Caught and fixed during the smoke test.)
- Calls `SELECT geojson, distance_m, duration_s FROM tiger.route_walking($1, $2, $3, $4)`.
- `tiger.route_walking()` returns a `MultiLineString` whose segments are NOT in travel order (each edge lands as its own LineString in graph order, sometimes individually reversed). Added `stitchLiveMultiLineString()` — picks the start segment closest to `(fromLat, fromLng)`, then walks endpoints with an EPS=1e-7° equality match (≈1 cm) and reverses segments as needed.
- After every `?source=live` call, the bundle is also routed for cross-check; logs `DIVERGENCE bundle=Xm live=Ym (Z%)` if they differ by >5%. None observed during the session — parity holds.

## P5 done — Web Directions UI in admin

### Flow

1. Click any POI marker → existing `PoiEditDialog` opens (unchanged path).
2. New **Directions** button at the bottom of the dialog (between Cancel and Save), enabled when the POI has finite lat/lng.
3. Click Directions → dialog closes, AdminMap fetches `/api/salem/route` from the **current map center** to the target, draws the polyline, and floats a panel top-right with:
   - POI name
   - Distance (m or km) + duration (min or h/min) + walking pace
   - Geometry-points + edge count
   - **bundle** / **live** source toggle (re-fetches and re-draws on switch)
   - × close button (clears the target)
4. Origin is captured at fetch time so panning afterwards doesn't re-flow the route.
5. Bounds fit on the FIRST render of each (target, source) pair only — re-fitting on every update fights the operator if they pan in to inspect.

### Polyline style (matches Android #22C55E)

- Outer "casing": 8 px wide, `#0f5132` (emerald-900), 0.45 opacity — gives legibility on both light and dark Witchy tiles.
- Inner stroke: 5 px, `#22C55E`, 0.95 opacity.
- Origin marker: small green CircleMarker at the captured map center.
