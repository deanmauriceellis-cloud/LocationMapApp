## Investigation 1 — Stale GPS 24h retention

**DB ground truth (at 20:20 local):**
- `gps_track_points`: 6,946 rows total
- Oldest row: 24.18 hours old (2026-04-09 20:09:51)
- **511 rows are older than 24h** — stale and not yet pruned

**Prune mechanism (`GpsTrackRecorder.kt`):**
- Retention: `RETENTION_MS = 24h`
- Cadence: `PRUNE_INTERVAL_MS = 1h`, piggy-backed on `recordFix()` calls
- Also fires once in `pruneStaleAtStartup()` from `SalemMainActivity.onCreate()`
- Query path (`getRecent24h()`) uses a `ts_ms >= now - 24h` cutoff, so the drawn polyline is always correct — the stale rows never render.

**Observed in logs:**
- 18:42:23 — startup prune deleted 416 rows
- 19:03:39 — startup prune deleted 20 rows (activity recreate from S110 churn)
- 20:09:50 — hourly piggyback prune deleted 1152 rows
- Between prunes and now (20:20), another 511 rows have aged past 24h but haven't been touched yet.

**Root cause:** the hourly cadence is the design, but it means the DB constantly holds up to 1 hour of stale data — more if the activity stops receiving fixes (screen off, background). Anyone querying the DB sees rows older than 24h even though the on-screen journey line is correct. The operator almost certainly observed this by querying the DB directly, which is the only surfacing path for the stale rows.

**Proposed fix (surgical, one constant):**
- Change `PRUNE_INTERVAL_MS` from `60 * 60 * 1000L` (1 hour) → `5 * 60 * 1000L` (5 minutes).
- Worst-case drift drops from ~1h to ~5min. Insert/delete overhead is negligible (prune is a single indexed DELETE).
- Not a feature change. No schema migration. ~1 line.

**Secondary observation (not a fix this session):** `pruneStaleAtStartup()` does `lastPruneMs = nowMs` before dispatching `runPrune`. Because the S110 lifecycle churn triggers `onCreate()` multiple times per minute, the hourly timer keeps getting reset. Not harmful (the startup prune still fires immediately each time), but worth noting — under churn the "hourly" prune never actually runs on the hour interval.

## Investigation 2 — POIs not going off

**What the log actually shows for the 19:18-19:27 downtown walk:**
- Entered downtown Salem at 19:17:14 (97 POIs nearby)
- 16 POIs narrated across 9 minutes:
  Universal Steel Lot, Grace Episcopal Church, The Witch House, Black Veil Shoppe, St. John Ukrainian Church, New Market, Bill & Bob's Roast Beef, and others
- **But 6 POIs fired `NARR-GEO: ENTRY` and got filtered out before speaking** — all with the same log signature: `NARR-QUEUE: Dequeue dropped N behind-user: <name>(<dist>m)`.

**The 6 dropped POIs:**
| Time | POI | Distance |
|---|---|---|
| 19:20:29 | Salem Athenaeum | 22m |
| 19:22:04 | Nathaniel Bowditch House | 21m |
| 19:22:04 | The Good Witch of Salem | 23m |
| 19:22:04 | Historic Salem, Inc. | 18m |
| 19:26:11 | **ChezCasa** | **9m** |
| 19:26:11 | Coffee Time Bake Shop | 19m |
| 19:26:31 | The Satanic Temple Salem Art Gallery | 23m |

**That's ~27% of ENTRY events silently dropped.** ChezCasa at 9 meters is the smoking gun — the user was essentially standing on top of it.

**Root cause — `pickNextFromQueue()` in `SalemMainActivityNarration.kt`:**
- S112+ added a "behind user" filter (`isPoiAhead`) that drops POIs outside a 90°-half-angle cone around the current movement bearing.
- The intent was to stop narrating POIs the user had already walked past. The effect is that when `lastMovementBearing` is noisy (user stopped briefly, slowed, or the GPS jitter between two close fixes pointed sideways), POIs within a few meters get dropped because they happen to land just outside the cone.
- At short distances (< ~20m), the user is effectively AT the POI and directional gating becomes noise — even a "behind" POI is something the user just walked past and would reasonably expect to hear about.

**Proposed fix (surgical, distance escape hatch):**
- In `isPoiAhead`, if distance to the POI is ≤ 15m (or similar short threshold), return true unconditionally.
- Equivalent: in `pickNextFromQueue`, bypass the `isPoiAhead` check when `dist <= NEAR_OVERRIDE_M = 15.0`.
- Preserves the intended behavior at medium distance (drop POIs you walked past blocks ago), fixes the near-zone false negatives.
- ~5 lines total.

**Alternative fix (not preferred):** widen `AHEAD_CONE_HALF_ANGLE_DEG` past 90°. This loosens the filter globally and reduces its value for POIs actually behind the user. The distance escape hatch is more targeted.

- Under 15m, the anchor stays put — the user is dwelling.

**Silence reach-out block in `initNarrationSystem`:**
- Replaced the single `findNearestUnnarrated()` call with a ladder:
  1. If `dwellPoisPlayed >= DWELL_MAX_POIS_PER_SESSION` → log `REACH-OUT HELD: dwell cap reached`, return. No further narration until the user moves away from the anchor.
  2. Otherwise try the current dwell radius.
  3. If empty, step up the ladder (`TIGHT → MEDIUM → WIDE → MAX`) and retry at each step. Logs `NARR-DWELL: EXPAND` for each step.
  4. On success, latch `dwellCurrentRadiusM = attemptRadius`, increment `dwellPoisPlayed`, log `REACH-OUT: <name> (<dist>m) dwellRadius=<r>m dwellPlayed=<n>/6`, then fire `triggerReachOut`.
  5. If the whole ladder is empty, log with the new 100m context and clear the highlight.

**`NarrationGeofenceManager.findNearestUnnarrated`:**

### Implementation — `SalemMainActivity.kt` `startWalkSim()`

**New constants (activity scope):**
- `WALK_SIM_DWELL_TRIGGER_M = 15.0` — matches `NEAR_ZONE_OVERRIDE_M` in the narration file, so any auto-dwell anchor is also past the behind-user filter escape.
- `WALK_SIM_DWELL_DURATION_MS = 25_000L` — 25 seconds. Room for the 5s silence gate + several reach-out ladder steps. Below the 30s GPS_STALE_THRESHOLD_MS so the staleness gate doesn't suppress reach-out.

**Walk sim loop rewrite:**
- Before the loop, load the full 817-point narration set once (`tourViewModel.loadNarrationPoints()`) for in-memory proximity checks. Handle failure by falling through with an empty list — the sim still works, it just can't auto-dwell.
- Track `dwelledIds: HashSet<String>` so each POI can only trigger ONE dwell per walk sim session (prevents the simulator from freezing on a return pass through the same cluster).
- On every step, after the 1s delay that advances the walker, scan for the CLOSEST un-dwelled POI within `WALK_SIM_DWELL_TRIGGER_M`. Picking the closest (not the first found) matters when a step crosses several overlapping radii at once — the anchor lands at the most relevant POI.
- When a trigger is found, enter a dwell loop: re-send `viewModel.setManualLocation(point)` once per second for 25 seconds. This is deliberate — not a bare `delay()`. Two reasons documented inline:
  1. `lastFixAtMs` stays fresh so the S110 staleness gate doesn't suppress reach-out on long dwells.
  2. `updateNarrationUserPosition` sees `moved < BEARING_UPDATE_MIN_MOVE_M` each tick, so `lastMovementBearing` stays frozen, the dwell anchor stays put, and the reach-out ladder engages identically to a stopped real walker.
- On dwell end, log `DWELL END at <name> (held Xs, resuming walk)` and continue the loop.
- On walk complete, log `Walk complete — N auto-dwell anchors triggered`.
- Cancellation is respected at every await — Stop button works instantly during a dwell.

## Test cycle 3 findings — **THE REAL ROOT CAUSE: sensor fires at 100 Hz**

Operator: "looks as bad as before"

Pulled the log. **Shocking discovery:** the Lenovo's MTK HAL delivers `TYPE_ROTATION_VECTOR` at **~100 Hz via SENSOR_DELAY_UI**, not the documented 60ms (~16 Hz).

Sample trace during idle (tablet at 75°):
```
22:32:38.299 HEADING-UP: skip hysteresis
22:32:38.308 ORIENT-RAW (+9ms)
22:32:38.310 HEADING-UP (+2ms)
22:32:38.318 ORIENT-RAW (+8ms)
22:32:38.320 HEADING-UP (+2ms)
...
```

**Every 10ms.** Every sample fires:
- `onSensorChanged` → rotation matrix → remapCoordinateSystem → getOrientation
- `GeomagneticField` lookup (cached but still invoked)
- Static mode detection (shortest angle delta)
- `ORIENT-RAW` log line (DebugLogger synchronized ring buffer + logcat + file sink write)
- `onUpdate` callback → `applyHeadingUpRotation`
- Source selection, EMA blend, hysteresis check
- `HEADING-UP: skip hysteresis` log line (same sink cost)
- `setMapOrientation` (cheap) + internal invalidate → map redraw

All on the main thread. At 100 Hz. **The main thread is saturated doing ~200 log writes per second plus redraws.** The UI thread can't keep up with real work (like actually rotating the map or processing touch events).

**Secondary observation:** even the actual rotation window (e.g., 22:32:06.666 → 22:32:06.673, the 90° rotation) shows only 7ms between `static mode OFF` and `static mode ON`. The rotation is detected as a near-instant jump from 75.9° → 165.9°. No intermediate samples captured. Either:
(a) Main thread was so backed up during the rotation that sensor callbacks got coalesced by the HAL
(b) The user's rotation was physically that fast (90° in <30ms = unrealistic for hand motion)
(c) The MTK HAL drops intermediate samples when the main-looper Handler backlog exceeds some threshold

Most likely (a) — the main thread backup causes HAL coalescing. Drains the very sample stream we're trying to track.

## Plan for S116 (next session)

**The core insight:** rotation smoothness is a main-thread saturation problem, not a sample-availability problem. The Lenovo sensor actually provides plenty of data; my code is consuming all the CPU processing noise and logging.

**Concrete fixes to try in order:**

1. **Cut log chatter at the hot path.** Remove `ORIENT-RAW` DEBUG trace. Remove `skip hysteresis` DEBUG trace. Only log at INFO on real state transitions (rotation detected, apply committed, regime changed, accuracy changed). 100 Hz → 0 Hz log writes when nothing's happening. This alone may solve it.

2. **Rate limit the apply path to ~33ms (30 Hz).** 100 Hz → 30 Hz apply rate. Every 3rd sensor sample drives a redraw. 30 Hz is smooth for display. But rate limit ONLY the apply — let the tracker keep receiving 100 Hz samples so static detection isn't starved.

3. **Move sensor processing to a background HandlerThread.** The `registerListener(handler)` parameter can point at a background looper. `onSensorChanged` computes azimuth + declination off the main thread. Only the `setMapOrientation` call hops to the main thread (via `post` or similar). This offloads the ~200 log writes and sensor math from the UI thread.

4. **Tune static mode detection to use TIME not SAMPLE COUNT.** With 10ms samples, 3 consecutive = 30ms which is way too fast to declare stability. Should be "no movement for 100-200ms". Switch from sample count to wall-clock duration.

5. **Optional: request `SENSOR_DELAY_GAME` (20ms) explicitly** to guarantee a known rate across devices instead of letting the HAL decide. Ironic here — we'd actually be slowing the sensor down for this device.

**Do not remove in the next pass:** the SNAP, the hybrid source, the declination correction, the configChanges fix, the motion tracker freeze, the dwell expansion — all of those are correct. Only the apply-loop hot path needs rework.
