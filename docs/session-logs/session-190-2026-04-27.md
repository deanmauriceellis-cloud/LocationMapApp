### Tour-leg routing ‚Äî diagnostics, then the actual fix

Operator: "tours are doing strange things‚Ä¶ and sometimes a leg takes a long
path to something close by. Make sure we have verbose logging."

**Phase 1 ‚Äî diagnostics (admin-tool layer only).** Added haversine helper +
`diagnoseLeg` in `cache-proxy/lib/admin-tours.js` returning
`{straight_m, routed_m, duration_s, detour_ratio, point_count, elapsed_ms,
suspicious}`. `compute-route` and `recompute-leg` now log per-leg lines like:

```
[AdminTours]   leg 5  Foo (12345) ‚Üí Bar (67890)  straight=45m routed=412m
               ratio=9.16 pts=22 dur=296s t=11ms ‚öÝ SUSPICIOUS
```

`suspicious` heuristic: ratio > 1.6, or straight < 200 m + routed > 800 m,
or straight < 100 m + routed > 400 m. Returned in the JSON response under
`leg.diagnostics` and `lastCompute.suspicious_count`.

`web/src/admin/TourTree.tsx` ‚Äî `handleComputeRoute` now `console.groupCollapsed`s
a tabular summary in DevTools and stores the response in `lastCompute`.
New amber call-out under the Walking Route header lists every suspicious
leg (clickable to highlight on the map). Skipped legs get a separate rose
call-out showing the reason.

**Phase 2 ‚Äî graph diagnosis.** Operator showed three skipped legs on
`tour_Density` ‚Äî 35‚Äì105 m straight-line distances, all "no route found".
Added `salemRouteDiag` helper to `cache-proxy/lib/salem-router.js` (no
algorithm change ‚Äî purely introspection): returns
`{from_node_idx, to_node_idx, from_snap_m, to_snap_m, same_node,
from_snapped_lat, ‚Ä¶}`. Wired into `admin-tours.js` so the skip-log + skip
JSON now report **why** the router gave up:

```
leg 5 NO-ROUTE  ‚Ä¶  straight=105m  cause=no path between snapped nodes ‚Ä¶
       from_node=#4699 (snap 3m), to_node=#12752 (snap 3m)
```

Diagnosis: every endpoint snapped within 3 m of a real graph node. Snap was
fine. The four nodes were in **different connected components** ‚Äî Charter
Cemetery has 7 OSM pedestrian features that all give each endpoint a fresh
vertex ID, and many of those endpoints are at the SAME physical lat/lng
(verified: 3 different node IDs at `42.5206924, -70.8918647`). So features
that physically meet at a junction are routing-disconnected.

**Phase 3 ‚Äî the build-pipeline fix.** Operator: "yeah, see what is going
on with the router/graph engineering, the web admin portal requires it to
map legs correctly." Saved `feedback_routing_build_engineering_allowed.md`
noting the scope override of `feedback_tour_routing_is_content_not_engineering.md`:
build-pipeline graph-data fixes are now allowed; runtime router algorithm is
still off-limits.

`tools/routing-bake/bake-salem-routing.py` ‚Äî two new passes added BEFORE the
existing S179 edge-split:

1. `merge_coincident_nodes(edges, nodes, tol_m=3.0)` ‚Äî buckets nodes on a
   coarse 11 m grid; for each pair within 3 m picks the lowest-id survivor
   (TIGER < OSM-offset < split-introduced); rewrites all edges; drops
   self-loops and parallel duplicates (keeping shortest).
2. `bridge_isolated_components(edges, nodes, tol_m=60.0, max_island=30)` ‚Äî
   union-find over the merged graph; for each component with ‚â§30 nodes,
   adds one synthetic walkable edge to the closest main-component node if
   within 60 m. Tagged `mtfcc='SYNTH_BRIDGE'`, `fullname='bridge {N}m'`
   for traceability.

Result on this graph: connected components 49 ‚Üí 46 (merge) ‚Üí 14 (bridge).
Bundle: 44,223 / 47,704 ‚Üí 43,087 nodes / 46,317 edges. `tour_Density`
re-compute went from `4 legs / 3 skipped` to `7 legs / 0 skipped` (later
9 / 1 after operator added more stops, with 4 suspicious ‚Äî leg 5 / 6 / 7
hop bridges between cemetery interior and perimeter, leg 10 routes around
the cemetery because stops 117‚Äì119 sit in a graph-empty corridor between
Charter St and Essex St).

`cache-proxy/server.js` exposes `deps.salemRouteDiag` to admin-tours.

No Kotlin changes ‚Äî `routing-jvm/.../Router.kt` treats `mtfcc` as opaque
metadata (verified: only loaded into `EdgesEntity`, never branched on),
so the new `SYNTH_BRIDGE` value is harmless on Android.
