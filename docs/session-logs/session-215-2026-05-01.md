## Edge-point Dijkstra (S214 Phase 2)

### Pre-work

`routeBetween(b, sIdx, tIdx)` is pure node-to-node Dijkstra. `routeBundle(b, fromLat, fromLng, toLat, toLng)` calls `nearestWalkableNode` to snap each end then routes node→node. `admin-tours.js compute-route` calls `deps.salemRoute(a.lat, a.lng, b.lat, b.lng)` ignoring the S214-persisted `salem_tour_stops.edge_id` / `edge_fraction` entirely. Result: every free waypoint corner-pulls to the nearer node at compute time, even though it's already snapped to a known edge in PG.

### Design (locked)

- **Same-edge fast path:** if both endpoints bound to the same edge, emit polyline-between-fractions with no Dijkstra.
- **Otherwise: 2-pseudo-source Dijkstra.** When src is `(edge_idx, fraction)`, seed BOTH endpoints of that edge with initial dist `fraction * L` (srcIdx end) and `(1-fraction) * L` (tgtIdx end). Run Dijkstra normally. When dst is `(edge_idx, fraction)`, the stop-set is both dst-edge endpoints with partial-cost map; pick whichever entry endpoint minimizes `dist[X] + dst-side partial`.
- **Geometry:** `snap-tail` (src-edge partial polyline from src snap to chosen exit endpoint) + Dijkstra path + `snap-head` (dst-edge partial polyline from chosen entry endpoint to dst snap).
- **Endpoint with no edge binding:** falls back to nearest-node snap. So POI-based stops (no binding — POI lat/lng is canonical) and free waypoints (edge-bound) can mix in the same tour without special handling at the call site.
- **Stale edge_id:** if the bundle doesn't have the edge_id (e.g. routing-bundle rebake renumbered it), fall through to lat/lng node-snap. Defensive.
- **Node-only routeBetween left untouched** — the Android-facing `/salem/route` endpoint keeps using it. Edge-point support is added as a sibling `routeBetweenEdgePoints(b, srcEP, dstEP)` + `routeBundleEx(b, srcStop, dstStop)`.

### What shipped

**`cache-proxy/lib/salem-router.js`** — added (~290 LOC):

- `_interpAlongEdge(b, edgeIdx, fraction)` — interpolated `[lat, lng]` at a fraction along an edge polyline, using the existing `edgeSegCum` / `edgeSegTotal` cache + binary search for the segment.
- `edgePartialPolyline(b, edgeIdx, fromFrac, toFrac)` — returns the `[[lat,lng],...]` polyline from `fromFrac` to `toFrac` (auto-reverses if reversed). Includes interpolated endpoints so the polyline begins/ends EXACTLY at the requested fractions, plus all intermediate vertex points strictly between.
- `_resolveEndpoint(b, ep)` — accepts `{edge_idx, fraction}` OR `{node_idx}` OR `{lat, lng}`; returns normalized `{node_idx?, edge_idx?, fraction?, snap_lat, snap_lng}`.
- `_dijkstraTwoSource(b, sources, stopAtSet)` — Dijkstra variant that takes a list of pre-relaxed source `{idx, cost}` pairs (instead of one) and stops once it has popped every node in `stopAtSet`.
- `_reconstructFromDijkstra(...)` — walks back the prev arrays to a known source endpoint, building the geometry + per-edge step list.
- `routeBetweenEdgePoints(b, srcEP, dstEP)` — same-edge fast path + 2-source Dijkstra + best-entry selection + tail/head/path geometry stitching.
- `_stopToEndpoint(b, stop)` — converts a stop object `{lat, lng, edge_id?, edge_fraction?}` to an endpoint spec, including `edge_id → edge_idx` lookup with an O(1) cache (`b._edgeIdToIdx`) built lazily on first use (~17K edges, scanned once).
- `routeBundleEx(b, srcStop, dstStop)` — public router taking two stop objects.

Module return adds `_routeEx`. `cache-proxy/server.js:198` plumbs it as `deps.salemRouteEx`.
