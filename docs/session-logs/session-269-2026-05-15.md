### Architecture revamp: pool ∩ proximity model

Operator pushed back: *"if a user goes to the default passport and clicks on a POI and gets directions, again how is it useful? The passport is both global ... for the 'Whole Tour Finished badge'. the default passport should have all the POIs by passport. When in a tour, we see only the tour. Confirmed via two `AskUserQuestion` clarifiers:
1. Recompute behavior: **wipes everything; manual edits persist until next wipe**.
2. Pool count: **exactly one global pool**.

Architecture became:
- **Global pool** = the single `default_salem_walking` filter row (categories + require_historical_narration + min_geofence_radius_m + year range). Auto-baked filter SQL.
- **Per-tour passport** = `pool ∩ proximity-to-polyline`. Computed explicitly via the dialog's new "Compute Candidates" button.
- Recompute is destructive (operator-confirmed "wipes everything"); manual ticks/unticks persist until next compute click.

**Backend**: extended `passport-walk-candidates` to load the global pool's filter parameters (categories, require_historical_narration, min_geofence_radius_m, year range) and AND them into the WHERE clause. Returns a `pool` block in the response describing what was applied + `candidate_count`. Falls back to "all POIs with non-empty historical_narration" if no global pool authored.

**Frontend** (`WalkPassportDialog.tsx` substantially rewritten):
- Opens showing the **saved state** (no auto-compute).

- "Compute Candidates" button → confirm dialog ("Wipe the current list (N POIs) and replace with pool ∩ proximity at R m? Manual edits will be lost.") → on confirm, replaces selectedIds with all pool POIs within radius.
