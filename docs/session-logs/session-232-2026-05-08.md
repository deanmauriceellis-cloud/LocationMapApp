### Matrix-tilt 3D prototype behind a debug toggle (curiosity)

Operator framed it as a curiosity ask. Built a one-file fake-3D wrapper.

**`TiltContainer.kt`** (new, ~95 lines): FrameLayout subclass that holds a tilt angle (0–75°) and builds a perspective `Matrix` via `setPolyToPoly` — top corners pinch inward and the top edge nudges down for foreshortening, simulating a forward-tilted camera. `dispatchDraw` concats the matrix; `dispatchTouchEvent` runs the inverse on a copied MotionEvent so map pan / pinch / POI taps still hit the correct underlying coords.

### Billboard markers via single-pass canvas trick (failed)

Operator wanted POIs to billboard upright instead of laying flat. Tried two implementations:
1. **`canvas.concat(invMatrix)` to undo the tilt locally** — wrong because matrix multiplication is non-commutative; the concat appends to the right of `parent * tilt * heading`, giving `parent * tilt * heading * invTilt`, which only equals `parent * heading` if tilt and heading commute (they don't). Labels rendered upside-down when facing south because the heading rotation wasn't cancelled.
2. **`canvas.getMatrix(snapshot)` + `setMatrix(identity)`** to snapshot the cumulative transform, project the marker's MapView pixel through it to get the final screen position, then draw at identity — clean in theory but `Canvas.getMatrix` is deprecated since API 16 because it isn't reliable on hardware-accelerated canvases. Markers either rendered at bogus positions or got clipped out at higher tilts.

Reverted `BillboardMarker.draw` to a thin pass-through (`super.draw`) so markers stay visible — they tilt with the perspective plane like before billboarding. ~18 sites of `Marker(binding.mapView)` had been sed-replaced with `BillboardMarker(binding.mapView)` across 8 files; left those in place to avoid churn (`BillboardMarker` is now a one-line subclass with no behavioural difference vs `Marker`).

### Wider-tiles two-pass attempt (failed)

Operator's bigger ask: "use a group of tiles around our current group of tiles so that we do have the map information we need" — fill the wedges with real Salem instead of dark purple. Pixel 7a context made the perf worry irrelevant, so we tried it.

Implementation:
- TiltContainer.onSizeChanged programmatically resized MapView to 1.5× width with `leftMargin = -extraSide` so it overflows symmetrically.
- `clipChildren="false"` on TiltContainer so the overflow could render.
- New `wedgeFillMatrix` mapping `(-extraSide, 0)..(w+extraSide, h)` → `(0,0)..(w,h)` — flat-compresses the wider MapView into the visible TC rect.
- Two-pass dispatchDraw: pass 1 with `wedgeFillMatrix` (fills the entire screen including wedges, flat), pass 2 with the original top-pinched `tiltMatrix` on top (overwrites the trapezoid area with perspective view).
- BillboardMarker skipped pass 1 via an `isInTiltPass()` flag on TiltContainer, to avoid duplicate icons at different screen positions.

What broke:
- **POIs disappeared in tilt mode.** With pass-1 markers skipped + pass-2 markers using the failed billboard logic (`setMatrix(identity)` on HW canvas), most markers rendered at the wrong screen positions or got clipped.
- **Strange division at horizon** — the seam where pass 1's flat-compressed transform meets pass 2's perspective trapezoid (at `y = topY`). The two passes show the same world content under different perspective projections, so the world content visibly jumps at the boundary line. Inherent to two-pass; can't be fixed without a single seamless transform.

Operator: "let's revert back to the good single-pass version to get back into a proper limited view."
