### Why a refactor was the precondition

`:salem-content` is a pure-JVM module (`org.jetbrains.kotlin.jvm`); `:core` is `com.android.library` and produces an AAR, not a JAR. salem-content can't depend on `:core` directly even though the routing classes inside `:core` (`Router.kt`, `RoutingBundle.kt`, `RouteTypes.kt`, `TurnByTurn.kt`) are pure Kotlin. The JVM mirror of the loader (`JdbcRoutingBundleLoader`) lived in `:core/src/test/`, not exposed.

## Post-P6 deploy on Lenovo: live-route gap discovery, OSM surgical ingest, content-data bugs surfaced

### Operator-driven testing on HNY0CY0W

Operator ran live "Get directions" from POI detail sheets across multiple destinations. Polylines drew via TigerLine instead of OSRM (per P6) — this part worked cleanly. **But the route stopped short of pier/peninsular destinations**:

| POI | TigerLine snap gap |
|-----|-------------------|
| Derby Wharf Light Station | 349.3m |
| House of the Seven Gables | 365m |
| Salem Maritime NHP | 195.5m |
| Pickering Wharf, Salem Common 2, others | 50–200m |

Cause: TIGER carries roads only — pedestrian-only walkways (piers, park footpaths, set-back museum entrances) aren't in the road graph. The router snaps to the nearest walkable TIGER node, which sits back on the street.

### Fix attempt 1: first/last-approach segments in `WalkingDirections.kt`

Added straight-line approach segments from user GPS to the start snap-node, and from the end snap-node to the destination POI, in `bundleRoute()` / `bundleMultiRoute()` / `routeResultToWalkingRoute()`. Distance + duration include the approach.

**Bug surfaced immediately**: `the_house_of_the_seven_gables_3` polyline drew a 365m straight line *across Salem Harbor* because the approach was unaware of water/non-walkable terrain. Operator screenshot confirmed.

### Fix attempt 2: cap approach at 60m

`APPROACH_MIN_M = 8.0`, `APPROACH_MAX_M = 60.0`. Above 60m the approach is skipped with a `WalkingDir` warning ("`Tail approach skipped: Xm gap likely crosses non-walkable terrain`"). Sub-60m approaches still draw — fixes normal set-back POIs (parking lots, friends_burial_ground at 55m, red_s_sandwich at 56m). Pier POIs end honestly at the snap node, no false claim of swimming the harbor.

This was deployed and operator verified the over-water bug was gone. But the routes still didn't *reach* Derby Wharf Light or Seven Gables — a 60m cap is by definition not 200–400m of pier.

### Investigation: why aren't the wharves reachable?

Found TIGER nodes 12.7m from the lighthouse but flagged `walkable_node=FALSE` — the surrounding edges have `mtfcc=P0001` (nonvisible administrative boundary) and `P0002` (perennial shoreline). These aren't roads at all, they're jurisdictional/coastline polylines that happen to touch Derby Wharf. **TIGER never carries pedestrian-only paths** — piers, boardwalks, park trails are out of scope.

Surveyed our internal sources:
- `massgis.openspace`: has the Salem Maritime NHP boundary polygon, no centerlines
- `massgis.trails / dcrtrails / biketrails / ldtrails / mad_trails`: 0 features on Salem's downtown wharves (focus is state DCR / bike infrastructure)
- NPS public datasets: searched `services1.arcgis.com/fBc8EJBxQRMcHlei`, `services2.arcgis.com/iq8zYa0SRsvIFFKz/.../NPS_Trails`, `mapservices.nps.gov`, `hub.arcgis.com`, `data.gov`, `public-nps.opendata.arcgis.com`. **NPS Trails system-wide has 0 features for Salem Maritime (UNITCODE='SAMA')**. SA-prefixed UNITCODEs in the dataset are SAFR/SAJH/SAMO — Salem Maritime isn't represented. NPS classifies the wharves as historic structures, not trails, and the structures layer (if any) isn't publicly published.

### OSM ingest — first attempt (broad, FAILED)

Authorized as a stopgap when no internal source covers the wharves. Used the existing `tools/tile-bake/data/sources/massachusetts.osm.pbf` (already on disk for tile bake) — strict build-time, no network call.

`ogr2ogr` extracted 3,610 features in the Salem bbox (`-70.91 42.50 -70.85 42.55`); 2,071 are pure pedestrian (`highway IN ('footway','path','pedestrian','steps')`). Wrote `cache-proxy/scripts/ingest-osm-pedestrian-into-salem-edges.sql` to topologically integrate via:

1. Reproject 4326 → 4269
2. ST_ClusterDBSCAN (eps=1.6m, then 5m) over endpoints to identify shared OSM nodes
3. LATERAL nearest-walkable-TIGER-node within 5m → reuse TIGER ID; otherwise allocate new ID at offset `10_000_000_000`
4. INSERT into `salem.edges` (mtfcc='OSMP') and `salem.edges_vertices_pgr`

**Result was a regression**: only 80 of 2,715 OSM clusters (3%) hooked into the TIGER backbone within 5m. The other 2,635 OSM clusters formed isolated graph components — `pgr_connectedComponents` reported **608 disconnected components** in the walkable graph (was 1 before the ingest). Tour-leg bake collapsed 68 → 11 because consecutive tour stops would snap to OSM-only islands and Dijkstra returned 0 hops between them.

Also found and fixed during this attempt:
- Two stuck `psql` backends from earlier mid-script kills holding row locks; cleared via `pg_terminate_backend`
- `mtfcc` column is `varchar(5)`, so `'OSM_PED'` (7 chars) errored — changed to `'OSMP'`
- Slow LATERAL query used `::geography` cast which defeated the GIST index; rewrote to use planar `the_geom && ST_Expand(...)` with post-filter geography distance

### OSM ingest — second attempt (surgical, SUCCEEDED)
