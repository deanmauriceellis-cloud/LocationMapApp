## Phase 9P.5 — DONE

### Implementation
Added `GET /admin/salem/pois/duplicates?radius=15` to `cache-proxy/lib/admin-pois.js` (~140 new lines). Architecture:

1. **Bbox-prefiltered self-join in SQL.** A `WITH all_pois AS (...UNION ALL...)` CTE projects (kind, id, name, lat, lng) from all three tables, filtering out soft-deleted rows. The self-join uses `(a.kind, a.id) < (b.kind, b.id)` for canonical pair ordering (each pair appears exactly once). The bbox prefilter (`ABS(a.lat - b.lat) <= radius/111000` AND `ABS(a.lng - b.lng) <= radius/60000`) cuts the cross-product before Haversine evaluation. The 60000 m/degree denominator for longitude is conservative (Salem is at 42.5°N where actual is ~82000) — overshoots without affecting correctness.
2. **Haversine in pure SQL** (no PostGIS): `6371000 * 2 * ASIN(SQRT(POWER(SIN(RADIANS(b.lat - a.lat) / 2), 2) + COS(RADIANS(a.lat)) * COS(RADIANS(b.lat)) * POWER(SIN(RADIANS(b.lng - a.lng) / 2), 2)))`
3. **JS-side cluster grouping via union-find with path compression.** Each pair from the SQL result becomes an edge; nodes are keyed by `${kind}:${id}`. After all unions, group by root. This produces *transitive* clusters: A-B and B-C and C-D all collapse into one cluster of 4 even if A and D are far apart (chain). This is the correct semantics for duplicate detection — chained POIs in a tight area are likely the same physical thing.
4. **Centroid + member distances** computed in JS via Haversine. Members within each cluster sorted by distance from centroid asc (closest first); clusters sorted by member_count desc (biggest dupes first).
5. **Validation:** `radius=0` and `radius=banana` both rejected with 400. `radius>100` silently capped at 100.
