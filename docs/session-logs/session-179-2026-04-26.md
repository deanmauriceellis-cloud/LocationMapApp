## P2 — Mid-block snap fix via bake-time edge splitting

Operator field test: "Get directions" to Historic New England's Phillips House (34 Chestnut St) drew a polyline that did not run down Chestnut St. Logcat from PID 15976:

```
WalkingDir: Tail approach skipped: 108m gap likely crosses non-walkable terrain
WalkingDir: Bundled route: 594m bundle + 18m lead + 0m tail = 613m, 438s, 26 pts, 2 turns
```

Diagnosis: TIGER edges run uninterrupted between intersections. Phillips House sits mid-block on Chestnut, so the router's snap-to-nearest-walkable-VERTEX picked the Hamilton/Chestnut corner — 95m up the block. The 108m straight-line tail to the front door was over the 60m approach cap and therefore suppressed; the polyline ended at the corner without entering Chestnut at all. Confirmed in PG: nearest walkable `salem.edges_vertices_pgr` row was 95.3m from the destination, while the nearest Chestnut St edge was 26m (mid-edge, no vertex).

Two fix paths were laid out for operator. Both selected:
- **Option 1 (data, this session): bake-time edge splitting.** Densify the graph by inserting walkable nodes mid-edge on every long edge.
- **Option 2 (router, next session): runtime mid-edge projection.** Project the destination onto the nearest edge and route to the projection point. Carry-forward as task #7.

### Option 1 implementation

`tools/routing-bake/bake-salem-routing.py`:
- Added `haversine_m`, `parse_polyline`, `encode_polyline`, `split_long_edges` helpers.
- New constants: `SPLIT_MIN_M=60.0`, `SPLIT_TARGET_M=40.0`, `SPLIT_NODE_ID_BASE=20_000_000_000`, `SPLIT_EDGE_ID_BASE=5_000_000_000`.
- After `fetch_edges` + `fetch_nodes`, `split_long_edges` walks each edge with `length_m > 60m`, computes cumulative haversine along the parent polyline, picks `n_pieces = max(2, round(total/40))` evenly-spaced split points, interpolates each split position when it lands mid-segment between geometry vertices, and emits a chain of sub-edges. Each sub-edge inherits parent `mtfcc` + `fullname`, preserves the relevant slice of the parent polyline (so render geometry still follows the road's curve), and gets `walk_cost == sub_length` matching the existing 1:1 convention. Synthetic node IDs above `SPLIT_NODE_ID_BASE` keep them disambiguable from TIGER (82xxx) and the S178 OSMP offset (10xxx).
- `source_summary` meta string updated with the split parameters so future debugging knows the bake parameters.
