## GPS dead zone diagnosis + fix (Salem real-world test)

Operator reported "GPS updates were taking 10-15 seconds" during a Salem test, initially attributing it to the Lenovo TB305FU being a junk device. Investigation:

### Hardware audit (Lenovo TB305FU, serial HNY0CY0W)

`adb dumpsys location` lifetime KPIs from the device:
- Device: Lenovo Tab M9 (TB305FU), MediaTek MT8786V/N (Helio G99 class), Android 15
- **No cellular radio** (`pm list features` has no `android.hardware.telephony` — Wi-Fi only)
- GNSS HAL: `MTK_MNLD_default,MNL_VER_default`
- 4 constellations active in fixes: GPS, GLONASS, BeiDou, Galileo
- L1-only — `Total number of L5 sv status messages processed: 0` (no dual-frequency multipath resistance)
- **TTFF mean: 6.49s, accuracy mean: 3.09m, location failure rate: 0.0%, 7355 location reports lifetime** — chip is solidly mid-tier, not "junk"
- Most recent GPS fix: 42.556683,-70.871567 (north Salem) hAcc=2.3m with 21 satellites tracked. The fix was 4d21h old at the time of the audit, suggesting the user's last in-the-field test was several days ago, not "today" — they were misremembering.
- During the audit: airplane mode OFF, Wi-Fi on, location_mode=3 (HIGH_ACCURACY = GPS+Wi-Fi+network), location enabled. Operator was NOT in airplane mode during their Salem walk despite their preference for airplane-mode testing — earlier session response had incorrectly assumed airplane mode and was retracted.

**Conclusion on hardware: not junk.** Two real handicaps for Salem walking: L1-only chip (multipath in urban canyon) and no cellular AGPS (slow recovery from fix loss). Both are software-mitigatable.

### App-side audit

Found Salem app's GPS pipeline:

1. **`core/src/main/java/com/example/locationmapapp/data/location/LocationManager.kt`** — wraps FusedLocationProviderClient, requests `Priority.PRIORITY_HIGH_ACCURACY`, configurable interval / minInterval. Default 5s/2s. Correct.

2. **`app-salem/src/main/java/com/example/wickedsalemwitchcitytour/ui/MainViewModel.kt:92`** — initial `onPermissionGranted()` call requests `60_000L / 30_000L`. So **first GPS request after permission grant is 60s/30s, not 5s/2s**. This is fix-#3 territory (chicken-and-egg with the adaptive interval logic) — left as a follow-up per operator scope.

3. **`app-salem/.../SalemMainActivity.kt:1255`** — adaptive interval logic in the location observer:
   ```kotlin
   val desiredInterval = when {
       (speedMph ?: 0.0) > 20.0 -> 10_000L
       narrationActive          -> 5_000L
       else                      -> 60_000L
   }
   ```
   At slow Salem traffic speeds (5-20 mph) with narrationActive, GPS is requested at 5s. Correct mode for the operator's actual scenario.

4. **`SalemMainActivity.kt:1267-1304`** — **the bug**: 100m static dead zone. Every GPS fix within 100m of `lastGpsPoint` is silently skipped from updating the marker, the map center, and `lastGpsPoint` itself. Narration geofences and the status line are still fed (correct), but the visible cursor is gated.

### Why 10-15s — the math

Operator clarified mid-investigation that they were **slowly driving Salem traffic, not walking** (~13-15 mph). With a static 100m dead zone:

```
update interval = 100 m / speed_in_m/s
```

| Speed | Cursor refresh |
|---|---|
| 10 mph (4.5 m/s) | 22 sec |
| 13 mph (5.8 m/s) | 17 sec |
| 15 mph (6.7 m/s) | 15 sec |
| 20 mph (8.9 m/s) | 11 sec |

Maps **exactly** to the operator's "10-15 second" report. The chip was producing 5s fixes correctly; ~70% of them were getting eaten by the dead zone before reaching the marker.

### Fix applied — `SalemMainActivity.kt:1267-1292`

Replaced the static `< 100f` threshold with a proportional dead zone:

```kotlin
val deadZoneMeters = (point.accuracy * 2f).coerceIn(5f, 100f)
...
} else if (!isManual && distanceBetween(lastGpsPoint, point) < deadZoneMeters) {
```

- **5m floor** — when GPS reports tight accuracy, dead zone is 10m, real movement >5m is never frozen.
- **100m ceiling** — worst-case multipath safety net, identical to old behavior under degraded conditions.
- **2× reported accuracy in between** — when multipath/canyon widens accuracy, the dead zone widens with it (so jitter is still absorbed) but only as much as needed.

Self-adapts across walking, slow driving, and L1-only multipath without mode-specific tuning.

| Scenario | Reported acc | Dead zone | Cursor refresh @ 6 m/s |
|---|---|---|---|
| Walking, good sky | 3-5 m | 6-10 m | 1-2 s |
| Walking, urban canyon | 15-30 m | 30-60 m | 5-10 s |
| Slow driving, good sky | 3-5 m | 6-10 m | 1-2 s |
| **Slow driving, downtown** | **10-20 m** | **20-40 m** | **3-7 s** |
| Highway @ 60 mph | 5 m | 10 m | 0.4 s |

### "How close to a POI do we need to be for it to announce?"

Researched the actual narration trigger geometry from `NarrationGeofenceManager.kt` and the live PG distribution:

### "At locations heavy in POIs, do we queue up the POIs?"

Yes — there are TWO queues:

1. **POI-level queue** (`narrationQueue: ArrayDeque<NarrationPoint>` in the activity, `SalemMainActivityNarration.kt:142`): holds POIs waiting to play. Insertion order is **priority-based**, not FIFO: `point.adPriority > queued.adPriority || (point.adPriority == queued.adPriority && point.priority < queued.priority)` — paying merchants jump the queue, then historical importance breaks ties.
2. **Segment-level queue** (`queue: ArrayDeque<NarrationSegment>` inside `NarrationManager`): for the currently-playing POI's TTS segments (short, long, transition). One POI at a time; segments stream via the TTS `onDone` callback.

The activity collects geofence events via `collectLatest` on the `_events` SharedFlow. Each emission calls `enqueueNarration(point, jumpToFront = false)`. Duplicates (same POI ID currently playing or already in queue) are silently filtered. When TTS completes and the state flips back to Idle, `playNextNarration()` polls the next POI; if the queue is empty, the silence reach-out timer kicks in (5s) and either plays anything queued during that delay or fires `findNearestUnnarrated()` to keep dead air at bay.

**Real-world issues in dense clusters** (flagged but not fixed):
- Queue can balloon to 5-8 items in places like the Witch House corner (Witch House + Black Veil + Witchery + Trials Memorial + Salem Visitor Center within 100m). At ~30 sec of TTS each, that's 3-4 minutes of committed back-to-back narration triggered by one moment of standing still.
- No "still relevant?" check at dequeue time — operator could be 3 blocks away when a queued POI finally gets its turn.
- Per-POI cooldown is 2 min, not per-cluster, so a cluster all fires once and then nothing fires for 2 minutes.
- Priority sort is set-once (no re-sort if a more important POI arrives mid-queue).
- No queue size cap (unbounded growth in a really dense area).

Recommended fixes (all flagged for future, none applied this session):
- **Stale-pop check** (cheapest, ~5 lines): when `playNextNarration` dequeues a POI, verify the user is still within `2× geofenceRadiusM` of it. If not, skip and pop the next.
- **Queue size cap** (e.g. `MAX_QUEUE = 4`).
- **Cluster batching**: treat 3+ simultaneous triggers as a single attention event, narrate the highest-priority one, dock-display the rest as "nearby".
- **Time-bound the queue**: drop POIs older than N seconds.
