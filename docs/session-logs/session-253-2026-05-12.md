**Fix 1 — TiltContainer side-wedge fill (`TiltContainer.kt`).** Diagnosed the small wedge as one of the two triangular regions left over from the tilt matrix: `setPolyToPoly` pinches the top corners inward by `pinch = 0.5·t·w` (at tilt=42° → t=0.467 → pinch ≈ 23% w), and the sloped left/right edges of the resulting trapezoid leave triangular gaps between the trapezoid and the screen for y ∈ (topY, h). EXTRA_TOP_FRACTION's vertical extension doesn't reach those triangles. Lateral extension does — extrapolating src x<0 (and x>w) through the perspective matrix fills the wedges. Required overhang per side to fully fill at the top of the wedge is `Δ = 0.5·t/(1−t)·containerW`. Implemented as `EXTRA_SIDE_CAP = 1.5f` (max overhang fraction per side) and a tilt-proportional `sideRatio` computed exactly the same way as `pinch`. New `extraSidePx` + `savedMapWidth` + `savedMapLeftMargin` fields mirror the existing vertical-extension bookkeeping. `applyMapExtension()` now sets `lp.width = containerW + 2·targetSide` with `lp.leftMargin = -targetSide` (symmetric per-side overhang keeps the MapView geometric center on the TiltContainer center, same as the vertical extension's symmetry). Short-circuit guard updated to compare both top AND side targets so tilt-degree-only changes (30→45→60) trigger re-extension correctly. AskUserQuestion offered four options (tilt-proportional exact, fixed conservative 1.5×, aggressive 3×, just shrink pinch instead); operator picked tilt-proportional exact. Built + uninstall+install on HNY0CY0W. Operator verified: side wedge gone.
