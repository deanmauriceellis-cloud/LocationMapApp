**Root cause — data, not code:** The fireflies are projecting *correctly* given the polygon coords from MassGIS. The problem is the polygon coords don't match where the OSM Mapnik basemap renders the cemetery. Charter Street Cemetery (Old Burying Point):
- MassGIS centroid: `(-70.89219, 42.52043)` — authoritative state survey, NAD83 (EPSG:4269), 10m inward buffer applied during extract.
- OSM Mapnik render position (back-calculated from operator's screenshot): roughly `(-70.898, 42.522)`.
- Disagreement: ~450m in lon, ~100m in lat.

When the operator's camera was at the OSM-rendered cemetery, the MassGIS Charter Street Cemetery polygon was off the right edge of the viewport — so all 30 fireflies for that polygon projected to x>1340 and were clipped. At Essex Street Cemetery (where MassGIS and OSM happen to agree closely), fireflies render correctly center-of-viewport — verified at `firstProj=(687, 290)` on `viewport=1340x684`, zoom 19.54.

**Verified at Essex Street Cemetery:** drawn=26 fireflies, firstProj near viewport center, matching the visible cemetery polygon on screen. Code is healthy.

**Open question for operator:** how to handle the MassGIS↔OSM disagreement, given the hard rule (`feedback_no_osm_use_local_geo`) that spatial data must come from local TigerLine + MassGIS, not OSM.

The mismatch: the green cemetery polygon visible on the Witchy basemap is rendered from OSM data via the `tools/tile-bake/` pipeline (planetiler reads the Geofabrik MA OSM extract and bakes it to `salem-vector.mbtiles` → MapLibre style → raster tiles bundled into `salem_tiles.sqlite`). Per the README, the basemap is "Built on top of MassGIS data + OSM (via OpenMapTiles) + MHC Inventory historic designations." So the cemetery shape on screen is OSM-origin even though it's locally bundled. Our firefly polygon library queries `massgis.openspace prim_purp='H'` directly, where the only record at that location is gid 27152 — a 1455 m² 4-vertex sliver in the wrong place.

Confirmed with: `psql -d tiger -c "SELECT * FROM massgis.openspace WHERE geom && bbox AND prim_purp='H'"` → only gid 27152 in the Quaker cemetery area.

Operator picked **B1+B2**: switch the firefly extract to prefer `massgis.landuse lu37_1999=34` polygons (cleaner "Cemetery" land-use class) when their area is between 30%–300% of the matching openspace polygon, fall back to openspace otherwise; and add a side-car JSON of hand-corrected overrides for cemeteries where neither MassGIS table aligns.

Implemented:
