## Parking Lot

> Ideas and future functionality captured during this session. Will be triaged at a natural break — each item goes to the master plan, STATE.md carry-forward, a memory file, or gets cut.

### P1 — Four App Modes

The app should have 4 distinct modes of operation:

1. **Main Tours** — Curated red line / yellow line tours (the Salem Essentials, existing tour engine)
2. **Generated Tours** — Custom tours built by the tour builder (time-budget, interest-based)
3. **Explore Salem** — Free-roam ambient mode with full POI view, narration triggers, pull-up detail cards
4. **Salem 1692** — *NEW.* A sub-application using SalemIntelligence's newspaper corpus (203 daily editions, Nov 1691–May 1693) and NPC profiles (232 historical persons, 3 tiers) to let the user explore the 1692 Salem Witch Trials as an interactive historical experience. Data source: `GET /api/intel/salem-1692/export` endpoint.

*This implies a top-level mode switcher in the app UX — not just tabs within the tour view.*

### P2 — Overlapping POI Spiderfy

When multiple POIs share the same location (or overlap at the current zoom level), they stack and become unselectable. Fix: implement a spiderfy/orbit pattern — overlapping POIs fan out into a circle around their true position, each connected by a thin line to a red dot at the actual coordinates. User can then tap each individual POI. Collapses back when deselected or zoom changes. This is critical with 2,190 POIs — Essex Street alone has dozens of businesses in the same building.

### P3 — Complete Menu/Navigation Overhaul

The current 9-dot grid menu and all navigation menus need a full redesign from the Salem tour perspective. This is not a tweak — it's a complete redo:

- **Graphics** — All menu icons/imagery need to be Salem-branded (witch city aesthetic, matching hero image style), not generic Material defaults
- **Layout** — Current layout is too basic/developer-functional. Needs to feel like a polished consumer product worthy of $4.99-$9.99
- **Structure** — Menu hierarchy should reflect the 4 app modes (P1), not the old generic LocationMapApp feature list. Every menu item should make sense from a Salem tourist's perspective, not an engineer's
- **Scope** — This touches the entire app chrome: nav drawer, bottom bar, settings, tour picker, POI categories, everything

*This is a Phase 10/11 item but the design thinking should start early. Ties directly to P1 (four modes) — the menu is how users switch between them.*

### P4 — Top-Layer UI Contextualization

The floating top-layer UI elements (cloud button, home button, and submenus like utility/find/etc.) are still generic LocationMapApp controls. They need to be entirely revisited and recontextualized for the Salem tour application — what does "home" mean for a tourist? What does "find" surface? What belongs in utility vs. being promoted to a primary action? These controls should feel native to the Salem experience, not inherited from a mapping SDK.

*Related to P3 — this is the map-overlay portion of the same navigation overhaul.*

### P5 — Intelligent POI Visibility & Narration Control

With 2,190 POIs the user can't see everything at once and shouldn't have to. Need an intelligent filtering system that lets users make smart decisions about what they want to see:

- **Group-level toggles** — Show/hide by category (Food, Witch Shops, Historic, etc.) with contextual groupings that make sense for a tourist, not a database admin
- **Fine-grained control** — Individual POI visibility and narration on/off beyond just category toggles
- **Smart defaults** — Context-aware: walking a tour? Show tour stops + nearby food/drink. Exploring freely? Show everything visible. In Salem 1692 mode? Show historical sites only.
- **Narration control** — Separate from visibility: user may want to see a POI on the map but not have it narrate. "I can see the restaurants but only tell me about the historic sites."
- **Density management** — At city-wide zoom, cluster or thin. At street-level, show everything relevant. Tie into P2 (spiderfy) for co-located POIs.

*This is the UX brain behind the 2,190-POI dataset. Without this, the map is either empty or overwhelming.*

### P6 — Top Ribbon as Contextual Information Bar

The top ribbon (weather/home/controls) is wasting premium screen real estate on buttons. Redesign it as a contextual information bar that gives the user useful awareness:

- **District awareness** — Show which Salem district they're currently in (Downtown, Derby Wharf, McIntire, etc. — we have 10 districts from SalemIntelligence)
- **Contextual info** — Number of nearby POIs, current tour progress, distance to next stop
- **Friendlier controls** — The buttons that remain should be warm and accessible, not utility-cold
- **Adaptive content** — What the ribbon shows changes by mode: tour mode = progress/next stop, explore mode = district/nearby count, 1692 mode = current date in the crisis timeline

*Ties into P3/P4 (navigation overhaul) and P1 (modes). The ribbon becomes the app's "you are here" awareness layer.*

### P7 — Narration Geofence Boundary Enforcement

Narrations are currently triggering for POIs well outside the intended boundary. The geofence trigger radius and/or the eligible POI set isn't properly clamped to the downtown Salem operating area. Need to enforce a hard boundary — no narration triggers outside the defined tour/explore zone. This is a bug, not a feature request.

*Likely a quick fix: check the narration trigger logic in `NarrationGeofenceManager` against the Salem bbox or a defined polygon. May also relate to the 589 hidden POIs (default_visible=false) leaking into the narration queue.*

### P8 — Orientation Change Destroys State

Rotating the device (landscape↔portrait) causes the Activity to recreate, losing current map position, zoom level, active tour state, narration playback, and falling back to default location. Classic Android lifecycle issue — need to persist and restore state across configuration changes via `onSaveInstanceState`, ViewModel retention, or locking orientation. This is a bug that will frustrate every real user.

### P9 — User Map Rotation / Bearing Control

Users should be able to rotate the map by changing its bearing — two-finger twist gesture to rotate, with a compass indicator showing current orientation and tap-to-reset-north. Currently the map is locked north-up. osmdroid supports rotation via `MapView.setMapOrientation()` — need to wire up the touch gesture and reconcile with the existing heading-up mode (which auto-rotates based on device compass). User manual rotation should override heading-up until reset.

### P10 — Onboarding / First-Run Experience

Tourist downloads the app at their hotel or Logan Airport. Currently there's no guided first-run flow. Need: welcome screen, interest picker (history? food? witchy stuff? all of it?), GPS permission grant with context ("we need this to guide your tour"), quick intro to the 4 modes, and a "Start Exploring" action that drops them into the right mode. This is the first 60 seconds — it determines whether they refund.

### P11 — Search UI

2,190 POIs with no search bar. User wants "Witch House" or "pizza near me" — there's no way to get there except scrolling the map. SalemPoiDao already has search queries built. Need a search UI: top-bar search field, results list with category icons and distance, tap-to-navigate. Should search across POI names, categories, and addresses.

### P12 — POI Detail View Redesign

The hero images we're generating need a home. The POI detail card/bottom sheet is the core content consumption screen — hero banner at top, narration text (short/medium/long from SalemIntelligence), hours, phone, website, "walk me there" button, category badge, district tag. This is where the user decides "do I go here?" — needs to be beautiful and functional, not a debug dump.

### P13 — Tour Resume / Session Persistence

Tourist closes the app mid-tour to take a phone call. Reopens. Current state is lost — tour progress, current stop, narration position, map view. Need to persist active tour state to disk (Room or SharedPreferences) and restore on cold start. Goes beyond P8 (rotation) — this is full process-death recovery.

### P14 — Favorites / "Want to Visit" List

Tap a star/heart on any POI to save it. View saved list. Optionally generate a custom tour from favorites (feeds into Mode 2 — Generated Tours). Basic trip planning feature that every tourist app has. Simple Room table + UI.

### P15 — Battery Awareness

GPS + geofence monitoring + map rendering + narration playback all day drains the phone. A tourist with a dead phone at 2pm has a ruined day. Need: smart GPS duty cycling (reduce frequency when stationary), low-power mode option, battery level indicator or warning, and documentation of expected battery impact. Real-world usability concern.

"I'm at the House of Seven Gables!" — share a POI card (hero image + name + short description + app link) to Instagram, Messages, WhatsApp, etc. Uses Android's share sheet. Free viral marketing and the primary organic growth vector for a paid app. Share content should be visually attractive (the hero images make this possible).

OSRM walking directions exist for tours but users expect to tap any POI and get walking directions from their current location. Google Maps muscle memory. The OSRM infrastructure is already in place — need a "Directions" button on the POI detail view (P12) that calculates and displays a route overlay.

SalemIntelligence discovered 744 event/menu pages across 604 businesses. October Salem is wall-to-wall events — parades, shows, festivals, haunted houses. Need an events feed: "What's happening near me tonight?" Filterable by category and distance. Time-sensitive content that drives engagement during peak season. Data source already exists in SI.

October evenings in Salem — dark streets, bright screen in your face. Need a dark UI theme (already partially there with the Salem aesthetic) AND dark map tiles. osmdroid supports tile overlays with color filters. Auto-switch based on time of day or manual toggle. ### P24 — Hero Image Prompt Architecture Overhaul

The current hero-image-gen pipeline needs a major review. Two fundamental problems remain:

1. **Interior vs exterior classification** — This is a first-class data decision, not a generation-time hack. Needs a `view_type` column (`interior` | `exterior`) on `salem_pois`, set during BCS import. Classification by category:
   - **EXTERIOR:** parks, cemeteries, memorials, plazas, wharves, piers, beaches, playgrounds, historic outdoor sites, amusement sites, monuments, gardens, commons, fields, parking lots
   - **INTERIOR:** restaurants, cafes, bars, shops, salons, offices, clinics, garages, gyms, studios, galleries, hotels, theaters, churches
   - **EDGE CASES needing manual review:** malls (exterior building but interior shopping), historic houses (exterior architecture but interior museum), tour operators (could be either)
   - SalemIntelligence should classify this based on entity_type + BCS category + description keywords, and expose it via the API so LMA inherits it during import

2. **Prompt quality doesn't do justice to Salem businesses** — The LLM-generated hero prompts from SalemIntelligence were biased toward exterior architectural descriptions. The scene descriptions we built as replacements are generic per-category. Neither approach captures what makes each Salem business special. Need to re-run the SalemIntelligence LLM prompt pass with a completely rewritten synthesizer prompt that:
   - Explicitly demands interior perspective (unless flagged exterior)
   - Focuses on the PRODUCTS and CHARACTER of each business, not the building
   - Uses BCS data (specialties, origin_story, owners, long_description) to mine entity-specific visual details
   - Outputs in a format optimized for Stable Diffusion (short, noun-heavy, no narrative)

*This is a cross-project effort: SalemIntelligence prompt rewrite + LMA generation pipeline update. The current images are good enough to ship but the Salem businesses deserve better representation.*

### P25 — SalemIntelligence API Feedback Loop for Hero Images

SalemIntelligence KB has rich BCS data (specialties, origin_story, owners, long_description, press_mentions, amenities) for every business — the problem isn't missing data, it's that the hero image synthesizer doesn't mine it deeply enough. The fix is at the SalemIntelligence API level:

- **Upgrade the hero image prompt endpoint** — the synthesizer should pull ALL BCS fields and translate them into visual terms (specialties → products to show, origin_story → unique character, amenities → interior features)
- **Add a feedback API** — LMA's generation pipeline should be able to report back which prompts produced bad images (exterior when should be interior, generic when should be specific) so SalemIntelligence can re-generate those prompts with corrections
- **Interior/exterior flag** — SalemIntelligence should classify each entity as interior/exterior based on its entity_type + category + description, and the hero prompt should be written accordingly
- **Prompt format optimization** — Current prompts are narrative paragraphs. Stable Diffusion works better with comma-separated keywords and short noun phrases. The synthesizer prompt should output in SD-native format.

*The KB has the answers. The API just needs to serve them in the right format for image generation. This is a SalemIntelligence project task, reported via OMEN.*

### P26 — Entity-Bonded Hero Images via SalemIntelligence

The hero images must be bonded to ACTUAL data about each business, not generic category templates. Every hero screen should visually represent real information from SalemIntelligence:

- **Hex sells handcrafted oils, incense, candles, gris-gris** → the image should show THOSE specific products, not generic "witchcraft shop"
- **Gulu Gulu serves crepes and craft beer in a bohemian setting** → show crepes and beer taps in a quirky eclectic space, not generic "cafe with espresso machine"
- **Derby Country Store has nautical gifts, ship models, hand-painted buoys** → show THOSE items, not generic "retail shop"

This requires a complete rework of the image generation pipeline:
1. SalemIntelligence provides entity-specific visual descriptions via an upgraded API (specialties, products, unique character, interior description)
2. The LLM synthesizer translates those specific details into SD-native image prompts
3. The generation pipeline uses those entity-bonded prompts, not category templates
4. Quality review compares the generated image against the source data — does it actually show what the business sells?

*Current V1 images are placeholders. V2 must make every image feel like it was illustrated by someone who actually visited the shop. SalemIntelligence has the data to make this happen — we just need to build the bridge.*

### P27 — Category/Subcategory Must Drive Image Tone and Authority

The category/subcategory should dictate not just WHAT objects are in the scene, but the TONE, WEIGHT, and AUTHORITY of the image:

- **District Court** → commanding, imposing, authoritative — the judge's gavel is massive, the bench towers over you, the scales of justice loom, the law books are stern and judging you, even the courtroom railing looks like it would hold you in contempt. The animated objects should be intimidating, not silly.
- **Children's daycare** → playful, colorful, chaotic — toys everywhere with mischievous grins, crayons running around, finger paintings judging your art skills
- **Fine dining restaurant** → elegant, sophisticated menace — wine glasses looking down their stems at you, silverware arranged with military precision and disapproval
- **Dive bar** → gritty, rough, hostile — beer bottles snarling, sticky bar top, pool cues that want to fight you

The animated objects should match the PERSONALITY of the business type. A courthouse doesn't have the same energy as a donut shop. The category drives the emotional register of the whole scene — the objects' attitudes should reflect the authority level and character of the place they live in.

### P28 — Every Hero Image Must Be Visually Unique

Current generation produces many near-identical scenes — every insurance office looks the same, every restaurant looks the same. With 1,295 images, visual repetition is immediately obvious when scrolling. Every single hero image needs to feel unique:

- **Composition variety** — vary camera angle, zoom level, focal point, object arrangement. Not every scene should be a straight-on shelf view.
- **Color palette variety** — even within the Halloween palette, shift the dominant hue per entity. One cafe is purple-dominant, another is orange-dominant, another is green.
- **Seed variation** — currently all images use random seeds which helps, but the prompts are so similar per category that the outputs converge
- **Entity-specific visual hooks** — this circles back to P26 (entity-bonded images). If SalemIntelligence knows Gulu Gulu has "mismatched vintage furniture and string lights" and Breaking Grounds has "industrial exposed brick and pour-over stations," those details should make the images visually distinct
- **Possible approach:** inject a per-entity seed modifier or color directive, or add a "unique visual element" field to the SalemIntelligence hero prompt output

*1,295 images where no two look alike is the goal. Current V1 has too many twins.*
