## 16:13 EDT — Warm-start TTS engine + splash-hold

Root-cause fix: process-scoped `SplashVoice` singleton. New file `app-salem/src/main/java/com/example/wickedsalemwitchcitytour/util/SplashVoice.kt`:
- `initEarly(context)` fired from `WickedSalemApp.onCreate` step 4 — TTS engine bind begins ~38 ms after process start.
- `speak(text, utteranceId, onFinished)` queues the request if engine still initializing.
- `UtteranceProgressListener.onDone` posts `onFinished` to the main thread.
- `shutdown()` called by SplashActivity on transition.
- State machine: NOT_STARTED → INITIALIZING → READY | FAILED; synchronized; main-thread callback via Handler(Looper.getMainLooper()).

SplashActivity simplified: no more fixed 5 s timer. `SplashVoice.speak(...) { launchMainActivity() }` drives the transition. Safety cap = 15 s via postDelayed so splash can never hang. `AtomicBoolean launched` gates launchMainActivity so onDone + safety-cap can't fire twice.

Verified end-to-end on Lenovo HNY0CY0W:

- 16:24:40.834 SplashVoice.initEarly (+38 ms)
- 16:24:43.866 TTS engine ready
- 16:24:45.982 speaking (flushed deferred speak)
- 16:24:50.428 onDone → transition
- Total splash hold ≈ 9.5 s; full welcome line audible.
