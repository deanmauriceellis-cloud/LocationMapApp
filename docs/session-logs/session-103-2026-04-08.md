### Recon (data shape)

Before writing the component, queried PG for the actual category vocabulary across the three POI tables (since the master plan and the next-session memory both said "kind тЖТ category тЖТ subcategory тЖТ POI rows" but I didn't trust that without checking the data):

| Kind | Column | Distinct values | Total rows |
|---|---|---|---|
| tour | `category` | 7 (`witch_trials`, `visitor_services`, `maritime`, `literary`, `museum`, `landmark`, `park`) | 45 |
| business | `business_type` | 19 (`shop_retail`, `restaurant`, `services`, `attraction`, `venue`, `other`, `public`, `park`, `historic`, `cafe`, `shop_occult`, `lodging`, `museum`, `tour`, `medical`, `shop_psychic`, `bar`, `attraction_haunted`, `tour_ghost`) | 861 |
| narration | `category` | 29 (`shop`, `restaurant`, `services`, `attraction`, `venue`, `other`, `park`, `public`, `historic_site`, `cafe`, `witch_shop`, `tour`, `public_art`, `lodging`, `museum`, `psychic`, `medical`, `witch_museum`, `bar`, `place_of_worship`, тАж) | 814 |

Total: 1,720 тЬУ (matches STATE.md / PG canonical count).

**Critical finding:** `salem_narration_points.subcategory` is NULL for all 814 rows. `salem_tour_pois.subcategories` is a JSONB list of refinements (not a single column suitable for grouping). The "kind тЖТ category тЖТ subcategory тЖТ POI" plan from the master plan / next-session memory is wrong about the data тАФ there's no third grouping level. Decided to ship a **3-level tree** (kind тЖТ category тЖТ POI rows) and document the deviation in the file header so a future session can re-add the fourth level if/when narration's subcategory column is backfilled.

Categories are NOT layer IDs from `PoiCategories.kt`. They are domain-specific OSM-style strings. Decided not to attempt a taxonomy lookup at grouping time тАФ just `groupBy` on the raw field. Each kind has its own vocabulary; overlapping names (e.g. `restaurant` in both business and narration) are scoped under their kind so no collision.

### Implementation тАФ `web/src/admin/PoiTree.tsx`

NEW file, ~330 lines. Layout:

1. **Types** тАФ `PoiKind`, `PoiRow` (open-ended index signature so the full row rides through to the edit dialog), `PoiSelection`, internal `KindResponse` and `TreeNode`.
2. **Constants & helpers** тАФ `KIND_LABELS`, `getCategory(kind, row)` (handles the businessтЖТbusiness_type / tour+narrationтЖТcategory split), `fetchKind(kind)` (URL: `/api/admin/salem/pois?kind=...&include_deleted=true&limit=2000`, throws on `!res.ok`, defensively coerces `lat`/`lng` to numbers), `buildTree(byKind, showDeleted)` (groups, sorts categories by descending count then name, sorts POIs by name).
3. **`useElementSize` hook** тАФ ResizeObserver-backed measurement of the scroll container, since react-arborist needs explicit pixel `width`/`height` for its virtualized list.
4. **`poiSearchMatch`** тАФ predicate for react-arborist's `searchMatch` prop. Matches POI nodes by case-insensitive substring on name; returns false for kind/category nodes (the library handles ancestor visibility automatically when a descendant matches).
5. **`PoiNode` renderer** тАФ flex row with disclosure caret, label (truncated), right-aligned count for container nodes, click toggles container or `select() + activate()` on leaf, deleted rows render greyed-out italic with "(deleted)" suffix.
6. **`PoiTree` component** тАФ state: `byKind`, `error`, `searchTerm`, `showDeleted`, container ref+size. `useEffect` loader: `await fetchKind('tour')` first (smallest, 45 rows тАФ sequential to fire the Basic Auth prompt exactly once), then `Promise.all([fetchKind('business'), fetchKind('narration')])` (parallel тАФ credentials already cached). `useMemo` builds tree data from `byKind` + `showDeleted`. Toolbar: search input, show-deleted checkbox, totals strip (`1,720 POIs (tour 45 ┬╖ biz 861 ┬╖ narration 814)`). Tree pane only renders the `<Tree>` once container size is known and rows are loaded; otherwise shows loading or error states.

### Notes / known footguns

- **3-vs-4-level tree.** Documented the deviation in the file header. The next session that touches this file should know: the spec said 4 levels, the data has 3, the file is structured so adding the 4th level back is a `buildTree()`-only change.
- **Auth flow assumption.** The sequential `tour` first тЖТ parallel `business`+`narration` pattern relies on the browser caching Basic Auth credentials between fetches. This is universal modern browser behavior but if a future session sees auth prompts firing 3x, the fix is to also serialize business+narration.
- **Lat/lng coercion.** Defensive `parseFloat` on the fetched rows in case node-postgres ever string-ifies DOUBLE PRECISION columns. Currently they come back as numbers; the coerce is a no-op but cheap insurance for the 9P.9 map view.
- **Container sizing.** `react-arborist` requires explicit pixel `width`/`height`. The `useElementSize` hook handles it. If a future drag-handle resizes the left pane, the tree will reflow automatically.
