### #11 GPS-OBS heartbeat investigation — FIXED + VERIFIED

**Carry-forward statement:** "Lenovo TB305FU quirk: system GPS delivers fixes but app's tour observer reports them stale. Same family as broken-motion-sensor issue. Investigate why fresh fixes aren't reaching `lastFixAtMs`."

**Diagnosis (live logcat on stationary Lenovo desk-test):**

System `LocationManagerService` was firing GPS + fused fixes at 1Hz, healthy. App subscribed at the initial `interval=60000ms minInterval=30000ms` and was receiving fixes — but only at ~27s cadence, right at the edge of the 30s stale threshold. The adaptive GPS interval picker (which should bump cadence to faster intervals when active) **never fired**, leaving the subscription stuck at 60s forever.

Root cause: the adaptive picker (SalemMainActivity.kt ~line 2974) sat **downstream of the stationary-freeze early-return** (line 2933 `return@observe`). On Lenovo, `MotionTracker.isStationary()` is permanently true (TYPE_SIGNIFICANT_MOTION never fires — memory `reference_lenovo_motion_sensor_broken.md`), so when the device is also slow/still (speed < 0.5 m/s, derived speed < 0.3 m/s, distance < radius), `stationaryFrozen=true` → `return@observe` → **picker never runs** → subscription stuck. Walks worked because speedMps > 0.5 disabled the freeze, but every dwell/stop re-stuck it.

**Fix:**

1. **Lifted the adaptive picker above the stationary-freeze gate.** It only reads `update.speedMps` + `tourViewModel.isNarrating()`, no visible-layer dependency, so the move is safe. Picker now runs on every fix regardless of freeze state.
2. **Bumped `GPS_STALE_THRESHOLD_MS` from 30s → 45s.** The idle-branch subscription is 30s/min 15s; 45s gives 15s headroom for momentary delays. S110's reach-out gate (the original "15-min-old positions cycling through downtown POIs" problem) is still well-protected — 45s is comfortably below that magnitude.

**Verification (90s burn-in on Lenovo, stationary, post-fix):**

