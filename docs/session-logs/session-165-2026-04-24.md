## 2026-04-24 — Fuzzy Find improvement

### Context
User reported Find dialog needs much improvement. Want: fuzzy matching, prioritized/relevance-ranked results, click → go to geocoord.

### What shipped
1. **`FuzzySearchEngine.kt`** (new) — multi-tier in-memory scorer against all 1,837 POIs:
   - Tier 1: exact full name → 10,000
   - Tier 2: name starts with query → 5,000
   - Tier 3: all tokens start name words → 3,000
   - Per-token: exact word (600), substring (400), prefix (250), Levenshtein ≤1/2 (120)
   - Secondary: subcategory, cuisine, description, historical note, address/district
   - Distance bonus as secondary tiebreaker (up to 300)
   - Levenshtein: 1 edit for words ≥4 chars, 2 edits for words ≥7 chars

2. **`FindViewModel.kt`** updated:
   - Preloads all POIs at `init {}` on background thread (instant search, no DB round-trip per query)
   - `searchPoisByName` now uses `FuzzySearchEngine.search()` instead of `salemPoiDao.search()` (LIKE %query%)
   - Fallback: if cache empty at search time, loads from DAO first

3. **`SalemMainActivityFind.kt`** updated:
   - Debounce: 1000ms → 250ms (snappy response)
   - Click on ANY search result → dismiss dialog + `animateTo(geocoord, zoom=18, 800ms)` + POI refresh after 900ms
   - Long-press on result → opens detail dialog (preserving the full detail view)
   - Applied to all three result lists: fuzzy search results, category results, favorites results
