### Architectural concern surfaced by the user
The user said: **"update in admin tool has to update both the database and where-ever they are sourced."**

**Current state of write-back:**
- `salem_narration_points` is canonical in PG (S98). Admin PUT/DELETE/move endpoints from 9P.4 write to PG. Bundled `salem-content/salem_content.db` is now stale relative to PG and will continue to drift.
- `salem_tour_pois` and `salem_businesses` are canonical in the bundled DB. **PG copies are EMPTY (0 rows each).** This means the admin tool's existing 9P.4 endpoints, when called against `kind=tour` or `kind=business`, will return 404 because the rows don't exist in PG. **The admin tool is functionally broken for those two kinds today.**

**The right fix (mirrors S98 pattern):**
1. **Migrate `tour_pois` and `salem_businesses` to PG** as a one-shot import, exactly like S98 did for narration_points. Source from the bundled `.db` file (use Python's `sqlite3` builtin, or a Node script with `better-sqlite3`).
2. **Phase 9P.C — Publish loop** (already mentioned in S98 notes): when the operator hits "Publish" in the Admin UI, regenerate `salem-content/salem_content.db` from PG so the next APK build picks up edits. This is the "where they are sourced" half of the user's requirement.
3. With (1) + (2) in place, PG is the single source of truth for ALL POI kinds, the bundled `.db` becomes a downstream artifact, and admin edits propagate cleanly into the next APK build. No two-way sync needed.

**Recommendation: do (1) NEXT SESSION, before starting Phase 9P.B Admin UI work.** Building the React admin UI on top of write endpoints that 404 for two-thirds of the POI kinds would be wasted effort.
