This pivots the S182 "operator hand-curates tour stops" plan into a different mode of hand-curation: operator picks waypoints in admin, an admin tool computes per-leg walking routes against the same on-device router the APK ships, persists those legs, and the app renders the saved polylines directly. No runtime tour routing in the APK; tour rendering becomes a static-content lookup.

This is consistent with `feedback_tour_routing_is_content_not_engineering.md` — the "engineering" part is one-time admin tooling, not runtime/graph work. The router itself is unchanged.

### Plan agreed with operator

Per-leg storage (not a single tour-wide polyline) so each leg can be independently recomputed, distance-tracked, and hand-edited later.

1. **PG: new table `salem_tour_legs`** — (tour_id, leg_order, from_stop_id, to_stop_id, polyline_json JSONB, distance_m, duration_s, router_version, manual_edits JSONB nullable, computed_at). PK (tour_id, leg_order). FKs cascade from salem_tours and salem_tour_stops.
2. **Cache-proxy: expose router on deps** — capture the salem-router module's return value (`_route`, `_bundle`) onto `deps` so admin-tours can call the bundle router in-process per leg.
3. **Cache-proxy: new admin endpoints** in `lib/admin-tours.js`:
   - `POST /admin/salem/tours/:tour_id/compute-route` — builds N-1 legs from N stops.
   - `GET  /admin/salem/tours/:tour_id/legs`
   - `POST /admin/salem/tours/:tour_id/legs/:leg_order/recompute` — single-leg refresh.
4. **Web admin** — Compute Route button on tour edit panel; legs render as colored polylines on the Leaflet map; per-leg list with distance/duration + recompute action.
5. **(Follow-up sessions)** Salem-content bake includes `salem_tour_legs` → `salem_content.db`. App `TourViewModel.computeTourPolyline()` reads baked legs and renders directly.
