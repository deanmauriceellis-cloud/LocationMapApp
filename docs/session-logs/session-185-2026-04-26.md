### Schema-bump landmines (debugged on-device)

Operator launched the first debug install and reported "circling icon when starting a tour" ‚Äî turned out to be `TourVM: Loaded 0 tours, 0 POIs`. Room had run `fallbackToDestructiveMigration` and dropped every row in every table on first open. Took two passes to fully diagnose:

**Landmine 1 ‚Äî pre-existing schema drift.** The asset's tables (created by `salem-content/create_db.sql` long ago) had `DEFAULT` clauses on most columns. Room's v10 codegen-generated schema has no DEFAULTs. Room's `TableInfo.equals()` compares `defaultValue` strictly ‚Äî mismatch ‚Üí fallback. Fix: new `cache-proxy/scripts/align-asset-schema-to-room.js` rewrites every Room-managed table using the exact canonical CREATE TABLE from `app-salem/schemas/.../10.json`, preserving content.

**Landmine 2 ‚Äî `PRAGMA user_version` was still 9.** Even after schema and identity_hash were correct, Room saw `user_version = 9` in the asset, treated it as a 9 ‚Üí 10 upgrade, ran fallback (no Migration registered), wiped everything. Hash matching alone is insufficient when version pragmas don't match. Fix: `align-asset-schema-to-room.js` also stamps `PRAGMA user_version = <version>` matching the schema JSON.

After both fixes, on-device DB opens cleanly: `tours = 1, tour_legs = 14, salem_pois = 2303`, `TourVM: Loaded 1 tours, 2303 POIs`.

This is a load-bearing rule for any future Room version bump in this codebase: bumping `@Database(version = N)` requires (a) regenerating the schema JSON via `kspDebugKotlin` with `exportSchema = true`, (b) updating the asset's `room_master_table.identity_hash` to the new value, (c) updating the asset's `PRAGMA user_version = N`, and (d) ensuring every Room-managed table's actual schema matches Room's canonical (no DEFAULTs). The align script handles (b)‚Äì(d) in one pass and should be run last in any publish pipeline.
