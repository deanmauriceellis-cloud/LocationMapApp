### Design decision (2026-05-09)

Two-pass dispatchDraw in `TiltContainer`:

- **Pass 1 (tilted):** strip `Marker` instances out of `mv.overlays`, run `super.dispatchDraw` with `tiltMatrix` concatenated → tiles + polylines + polygons + WickedAnimationOverlay (water/fireflies/grass) all render with the perspective shear (correct: ground-painted feel for everything that belongs on the surface).
- **Pass 2 (upright):** canvas matrix back at TiltContainer identity. Iterate stripped markers: `mv.projection.toPixels(geoPoint)` → flat mapView-local px → add `(mv.left, mv.top)` for container coords → `tiltMatrix.mapPoints` → post-tilt screen px. Stamp the icon UPRIGHT at that pixel with depth-based scaling.

**Sprite handling:** SpriteOverlay short-circuits its own draw via a `tiltActiveSupplier` callback when tilt > 0; TiltContainer drives a parallel `drawBillboarded` pass that uses osmdroid's rotation-aware projection + the tilt matrix. Same upright-with-depth-scale treatment as markers. Sprites render BEFORE markers in pass 2 (operator z-order rule: roads/POIs render OVER animations).

**Operator-confirmed scaling preference:** close POIs render 2× their flat-mode size, distant POIs scale down toward the horizon. Linear falloff between `BILLBOARD_NEAR_SCALE = 2.0f` (foreground / `dstY = h`) and `BILLBOARD_FAR_SCALE = 0.6f` (horizon / `dstY = topY`).

**Touch hit-testing untouched:** the existing `dispatchTouchEvent` inverse-transforms taps back to flat container coords; markers stay in `mv.overlays` at their flat lat/lng; pass-2 only changes pixels, not where Markers think they live.

At z16 same code paths fill correctly. Suspect osmdroid's internal `MapView.dispatchDraw` clips to its own `getWidth()/getHeight()` regardless of layout-params extension.
