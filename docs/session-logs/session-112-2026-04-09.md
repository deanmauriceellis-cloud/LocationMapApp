### Finding 1 — POI announcement starvation has a precise mechanism: APPROACH events consume the ENTRY cooldown

**Code:** `app-salem/src/main/java/com/example/wickedsalemwitchcitytour/tour/NarrationGeofenceManager.kt`

In `checkPosition()` lines 207-218, the APPROACH branch sets `cooldowns[point.id] = now`:

```kotlin
distanceM <= approachRadius -> {
    if (!isOnCooldown(point.id, now)) {
        cooldowns[point.id] = now   // ← BUG: APPROACH consumes the entry cooldown
        approachCount++
        _events.tryEmit(NarrationGeofenceEvent(type = APPROACH, ...))
    }
}
```

**Sequence in a dense cluster (real numbers from the drive):**
1. User is 50m from POI X. Entry radius = 30m, approach radius = 60m (= 30 × APPROACH_MULTIPLIER 2.0).
2. GPS fix arrives → APPROACH branch fires → `cooldowns[X] = now`. POI X is now on a 2-minute cooldown.
3. 5-10 seconds later, user has moved closer. New fix at 25m → ENTRY branch evaluated → `isOnCooldown(X)` returns TRUE → `skippedCooldown++` → **no narration triggered for POI X.**

The 2-minute COOLDOWN_MS (line 75) is meant to prevent same-POI rapid repeats. But because APPROACH and ENTRY share the same cooldown map, APPROACH "burns" the cooldown that ENTRY would have used.

**Evidence in the log (process 6067 NARR-GEO SUMMARY lines):**

| Time | Position | Nearby | Approaches | skippedCooldown | Narrated | Entries |
|---|---|---:|---:|---:|---:|---:|
| 18:24:56 | 42.52385,-70.89699 | 91 | 1 | 0 | 0 | 0 |
| 18:28:04 | 42.52131,-70.89884 | 111 | 6 | 1 | 1 | 0 |
| 18:29:09 | 42.52144,-70.89740 | **176** | 6 | **8** | 1 | 0 |
| 18:30:09 | 42.52038,-70.89558 | **244** | 0 | 3 | 1 | 0 |

`skippedCooldown` count is direct evidence — these are POIs the user moved INSIDE the entry radius, which would have triggered narration but were blocked because the cooldown was already set. Since `narratedAt` is the in-memory dedup map (not in cooldowns), and ENTRY events set BOTH `narratedAt` AND cooldowns, a `skippedCooldown` (without a corresponding `skippedNarrated`) can ONLY come from a previous APPROACH event having set the cooldown.

**Total drive damage:** approximately **30 POIs were silently dropped** by this bug across the 25-minute downtown Salem drive (cumulative skippedCooldown counts where the user was actually inside an entry radius). Combined with the 13 narrations that did fire, the user should have heard ~43 POIs instead of 13. (The remaining ~200 POIs the operator wanted to hear are blocked by other issues — primarily, geofence radii being smaller than the GPS sample-distance at typical urban driving speed.)

**The fix:** remove `cooldowns[point.id] = now` from the APPROACH branch. APPROACH is just a UI dock-update event; it should NOT consume the entry cooldown. ENTRY events should be the only thing that sets the cooldown. APPROACH still emits its event to the dock; the dock can deduplicate at its own layer if needed.

### Finding 2 — POI no-repeat window: only ONE 12h constant exists in the narration system; the "24h" the operator was thinking of is unrelated

**The 12h constant** is `NarrationGeofenceManager.kt:49`:
```kotlin
private val REPEAT_WINDOW_MS = 12 * 60 * 60 * 1000L  // 12 hours
```

**The 24h constant** is in a completely different subsystem — `GpsTrackRecorder.kt:148` `RETENTION_MS = 24 * 60 * 60 * 1000L`. That is the **GPS journey rolling-window retention** (how long the polyline keeps the user's path). It is NOT a POI repeat window. **We should NOT change this.**

So changing the no-repeat window 12h → 1h is a one-line change to `REPEAT_WINDOW_MS` plus a comment update plus the doc comment on `narratedAt` (line 45) and `isNarrated` (line 319).

**Note on impact:** since `narratedAt` is in-memory only (no SharedPreferences mirror), the 12h window only matters within a single process lifetime. Process death clears the dedup map and any narrated POI becomes eligible again. So the change from 12h → 1h is most significant during long single-session use (a 90-minute walking tour, or a long drive). Within the operator's 25-min drive, all 817 POIs were eligible from the start; reducing the window will only help on a return-trip pass (e.g., looping back through downtown Salem on the way home).

### Finding 3 — GPS journey line "disappears after N minutes never to come back"

**The mechanism is structural, not a bug per se: GPS recording is bound to the activity's STARTED lifecycle state.**

In `SalemMainActivity.kt:1483`:
```kotlin
viewModel.currentLocation.observe(this) { update ->
    ...
    gpsTrackRecorder.recordFix(...)        // Writes to user_data.db
    gpsTrackOverlay?.let { polyline ->
        polyline.actualPoints.add(...)     // Appends to the visible line
        if (binding.mapView.overlays.contains(polyline)) {
            binding.mapView.invalidate()
        }
    }
    ...
}
```

LiveData observers fire ONLY while the lifecycle is at-least-STARTED (Android default). The moment the activity goes to onPause/onStop (screen off, app backgrounded, switched to another app), the LiveData observer is suspended. **No fixes are recorded to DB. No fixes are added to the polyline.** The LocationManager still emits fixes, but they go nowhere from the recorder's perspective.

**Evidence in the log:**

- **18:43:58.751** `LIFECYCLE: onPause changingConfig=false finishing=false` — operator put down the tablet at home
- **18:43:58.844** `LIFECYCLE: onStop` — activity stopped (NOT destroyed)
- **NO `GPS-OBS fix lat=...` lines from 18:43:58 to 19:13:59** — the LiveData observer is suspended for the entire 30-minute window
- **19:13:59.292** `LIFECYCLE: onResume` — operator opened the app again
- GPS fix lines resume at 19:14:01 with `gap=2032ms` — fresh fix from the chip, but the 30-min window in between is permanently lost

The polyline still exists in memory (the activity wasn't destroyed). When the user opens the app again, the line is what it was at 18:43:58 — the full drive. New fixes after 19:13:59 will append, but THE GAP IS NEVER FILLED because no fixes were recorded during the pause period.

**The operator's "after 5 minutes" complaint** is consistent with this mechanism. The exact 5 min in their report is probably from an earlier drive that was interrupted by a screen-off or app-switch event. The line stops growing the moment the activity is paused. Reopening the app does not retroactively recover the GPS data from the pause window — that data was never sampled by the activity-bound observer.

**Secondary finding: 10 lifecycle recreates in 6 minutes (18:09:31 → 18:15:41) with `changingConfig=true`.** Each cycle destroyed and recreated the activity, briefly losing GPS fixes during the 1-2 second tear-down/rebuild. Net: ~57% of expected GPS fixes were dropped in this 6-minute window (76 → 106 historical points across 10 cycles, vs ~70 expected at 5-second GPS rate). Trigger of the cycles is unclear — `changingConfig=true` implies a system configuration change (orientation, dark/light mode, density, multi-window, etc.). Could be a Lenovo TB305FU device quirk or the operator wiggling the tablet in the car. Worth investigating but not the primary line-disappearing bug.

**Tertiary finding: the user_data.db was completely WIPED between processes 2315 (17:35:51, 647 historical points) and 3581 (17:46:21, 0 historical points).** Possible cause: Room schema migration with `fallbackToDestructiveMigration` (S110 added schema v1→v2). If the operator installed an updated debug APK between drives, the DB wipes per the configured migration policy. The operator's previous trip data was lost. This is documented behavior in the S110 commit message but is worth noting since it explains why the line might appear "shorter than expected" after an APK reinstall.

**Fix options for the disappearing-line bug:**

1. **Cheapest (partial):** Pull the GPS recording out of the activity's LiveData observer entirely. The recorder is already a Hilt `@Singleton` (process-scoped). Wire it to subscribe directly to the LocationManager source via a `@Inject` of the location source. The recorder no longer depends on the activity lifecycle. The visible polyline still depends on the activity (since osmdroid's overlay is per-MapView), but the DB-side journey log captures everything the LocationManager produces while the process is alive.
   - **What's still lost:** GPS during process death (system kills the app entirely). For that you'd need a foreground service.
   - **What's fixed:** GPS during onPause/onStop (screen off, backgrounded, switched apps).

2. **Correct (Android standard):** Foreground service for GPS tracking. Survives backgrounding indefinitely. Requires `FOREGROUND_SERVICE` permission, a persistent notification, and a service class. Standard pattern for fitness/navigation apps.

3. **Cheap and partial (workaround):** When the activity resumes, run a "catch-up" pass that fetches the LocationManager's last known location and inserts it as a single point. This DOES NOT fill the gap — it just shows the user where they ended up. The intermediate path is still lost.

4. **Hybrid:** Foreground service only when the user explicitly enables "track my drive" mode. Default off. Operator-controlled.

### Architectural changes that landed in S112

- **Narration queue is now tier+density+bearing-aware.** The dequeue picks the closest in-front POI within the highest non-empty tier, with the discard radius adapting to local POI density (40m → 30m → 20m as density increases).
- **The dock is the play queue, visible.** Top of dock = next to play. Same filter, same sort. Empty queue = empty dock.
- **APPROACH events no longer consume the entry cooldown.** They emit dock-update events without touching the cooldowns map.
- **Walk-sim is now a transport-only simulation.** No more radius inflation; the simulator just feeds synthetic GPS fixes at walking pace. Narration logic treats it identically to real GPS.
- **Cold launches reset the narration history.** `savedInstanceState == null` branch in `onCreate` calls `resetSession()`. Config-change recreates (orientation, dark mode) skip the reset and preserve the S110 singleton state.
- **Glowing highlight ring** marks the currently-narrated POI on the map. 5m/9m gold halo, pulsing alpha animation. Auto-replaces on each dequeue, auto-clears on queue drain.
- **User trigger ring debug overlay** shows the three discard tiers (20m red / 30m amber / 40m green) as outline circles centered on the walking icon. Provides at-a-glance "what's about to trigger" visibility.
