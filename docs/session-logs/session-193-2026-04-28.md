## Demo-bug fixes (mid-session, after operator reported newspaper repeats + TTS interruptions)

### Root cause

`NarrationManager.onDone` (line 108) fired for **every** TTS utterance —
including each chunk sub-id (`-c1`, `-c2`...) and each silent-pause filler
(`-p1`, `-p2`...). It unconditionally cleared `currentSegment` and called
`playNext()`. After chunk 0 of a long narration finished, the app-level
queue was already empty → state flipped to **Idle** while the TTS engine
was still playing chunks 1..N from its internal queue.

Cascade:
1. Mid-narration state flips to Idle
2. Silence-fill observer thinks the queue is empty → fires a 1692
   newspaper after a 3 s delay
3. Newspaper enqueue → `tts.speak(..., QUEUE_FLUSH, ...)` wipes the
   queued POI chunks from the engine
4. Headlines repeat: `historicalHeadlineQueue.advance()` runs *after*
   the 3 s delay, so a second silence-fill in the same window grabs the
   same headline

### Fixes shipped

**Fix 1** (`NarrationManager.kt:108-148`) — chunk-aware `onDone` /
`onError`. Treat the segment as finished only when `utteranceId ==
currentSegment.id` (the LAST chunk, by chunker design). Intermediate
chunk + silent-pause ids return early without touching state.

**Fix 2** (`SalemMainActivityNarration.kt`, 5 sites: heartbeat +
`runSilenceFill` 4-way) — call `historicalHeadlineQueue.advance()`
**before** the 3 s delay. A second silence-fill or the heartbeat firing
during the delay window can no longer re-fetch the same headline.
Headline is lost-on-abort — intentional; better than re-firing.

### Verification (live device)

Log lines confirm working:
- `TTS chunk done: ...-c0 (mid-segment)` / `-p0 / -c1 ...` — chunks
  recognized as mid-segment, no premature Idle.
- `narrationState → Speaking(...)` stays for the full multi-chunk POI
  narration, only flips to Idle when the bare segment id finishes.
- `NARR-HEARTBEAT: TICK` followed by single `FIRE` → `advance` → `delay`
  → speak; no double-fires.

Operator confirmed Oracle newspapers now firing correctly after the 5-min
heartbeat interval elapsed post-reinstall.

## Mid-session field-bug fixes (after the dormant historical-tour wiring shipped)

### Walk-sim FAB race + synthetic-route fallbacks (`c2277aa`, `3f578a4`)

Operator: "FAB walk seems to lost its mind." The walker was marching a
21 km Downtown Salem perimeter loop instead of the active tour's
`tour_legs` polyline. Two bugs in `SalemMainActivity.startWalkSim`:

1. **Race**: when `tourState` was Idle/Loading, the auto-call
   `tourViewModel.startTour("tour_WD1")` is async. The very next line
   evaluated `activeTour` while the tour was still Loading, so
   `activeTour` stayed null and the route selector fell through to
   `loadDowntownRoute()`.

2. **Synthetic-route fallbacks**: the route selector had four layers —
   baked tour_legs polyline → legacy bundled JSON
   (`TourRouteLoader.loadAllRoutePoints`) → straight-line stop-coordinate
   stitch → 1776-point Downtown perimeter. A failed/empty tour silently
   substituted a wrong route.

**Fixes:**

- `c2277aa` — `withTimeoutOrNull(3 s) { tourState.first { it is Active } }`
  in the auto-start branch; rebind `activeTour` from the new state on
  success. Symmetric wait added to the Paused → resume branch. On timeout
  the FAB used to fall back to Heritage Trail JSON; that's also gone now.

- `3f578a4` — operator: "there is no fallback, what we have as tours is
  definitive." All synthetic fallbacks removed. New behavior:
  - no active tour → toast "Pick a tour first", abort
  - empty `tour_legs` polyline → toast "Tour 'X' has no route — author
    it in admin", abort
  - auto-start timeout → toast "Tour didn't start — try again", abort

  Saved the rule as a hard memory:
  `feedback_authored_tours_are_definitive.md`. Updated `MEMORY.md`
  index. Future sessions can't reintroduce a fallback path without
  contradicting this rule.

### Salem 1692 newspaper queue: no loop-back, no repeats (`7d15b1c`)

Operator: "Why do we hear repeating narration of the Salem Newspaper?
... I don't mind having all the information, but a single newspaper
item should not be repeated."

Two repetition causes:

1. `pollNext()` looped back to `idx=0` once the 202-dispatch corpus
   exhausted — the same dispatches replayed in order from the start.
2. Uninstall wipes `SharedPreferences`, so a reinstall (which we did
   several times this session for the chunk-aware fix and the FAB-walk
   fix) reset the index to 0. Operator heard dispatches 2-5 then 1
   again post-reinstall, perceived as repeating.

**Fix** (`HistoricalHeadlineQueue.kt`):

- `pollNext()` now returns `null` when `idx >= list.size`. Queue stays
  silent until `reset()` is called explicitly.
- Bumped prefs key `newspaper_next_index_v2` → `_v3` so any in-progress
  walk under the loop-back rule starts cleanly at index 0.
- Order kept chronological per operator's clarification ("all the
  information"). A speculative length-desc sort I'd shipped between
  the two operator messages was reverted before commit.

### Field-test field bug — sheet read-through bypass (`1992606`)

narrations. Logs showed the geofence path's `getNarrationForPass` was

Root cause: `PoiDetailSheet.queueSheetReadThrough` builds a 4-segment
auto-read (intro + shortText + aboutText + storyText) that bypasses
`NarrationGeofenceManager.getNarrationForPass` entirely. Even with
`historicalNarrationMode=true` on the geofence side, opening a POI

  paused tour.isHistoricalTour. Public surface for the UI layer.
- `PoiDetailSheet.queueSheetReadThrough`: when active and the POI has
  `historical_narration`, queue `intro + historical` only and skip
  short/about/story.
