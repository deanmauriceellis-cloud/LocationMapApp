#### Findings

**1. [HIGH] TransitViewModel main-thread block — 118 slow-dispatches averaging 1.1s (p95 1.2s, max 1.36s) from `fetchMbtaBuses` at line 94.**

Every `_mbtaBuses.value = it` emission on `.onSuccess` blocks main for ~1s when the bus list is large. Spike is correlated with fleet size: at 07:00 MBTA returned **616 vehicles** and we got 51 slow-dispatch warnings in that hour alone. Outside the 06:00-09:00 window (rush hour, dense fleet), dispatches stayed under the Looper warning threshold.

Cause is almost certainly the StateFlow → Compose recomposition of the map overlay with hundreds of bus markers. The network + parse already happen on `Dispatchers.IO`; the cost is the Compose pass.

**Fix directions (pick one or combine):**
- Diff the incoming bus list against the prior emission and only emit if changed — reduces needless recomposes (a 60s tick of identical data still triggers StateFlow emit).
- Throttle map-bound marker updates to once per ~2-3s max — decouple transit fetch cadence from UI redraw cadence.
- Downshift to bus summary (count + bounding box) for overview; full fleet only for zoomed-in view. This is a real UX gain, not just log hygiene.

**Occurred:** 717 MBTA bus fetches in 12 hours, 60/hour, gap ≈ 60s; 2 failures, 715 successes. MBTA goes directly to MBTA API, not through cache-proxy — that's why it kept working while cache-proxy was down.

**2. [MEDIUM] GPS-OBS stale-fix heartbeat log spam — 1,208 W-level lines over 12 hours (1 per 30s while stale).**

Pattern: `HEARTBEAT still-stale — last fix 34,921s ago`. Each heartbeat fires every 30s regardless of how stale GPS has been. On the workstation the Lenovo sat indoors with no real GPS for 9.7 hours — that produced 1,200+ `W` warnings.

**Fix:** backoff schedule. First stale-transition logs once at `W`. After (say) 10 min stale, drop subsequent heartbeats to `D` or switch to a 5-min cadence. Or just stop repeating the "still-stale" line once it's already been logged once at the same state.

**Occurred:** 1,208 W lines + 5,922 I-level companions = 7,130 GPS-OBS log lines in 12 hours. This is the single noisiest tag.

**3. [MEDIUM] Cache-proxy circuit-breaker log spam — ~560 warnings across 4 ViewModels + LocalServerHealth.**

Pattern while cache-proxy is down: every failed request logs a `W`-level "circuit-open, fast-failing for Ns more" line, even though the circuit is already known-open. Per-endpoint counts:
- `Webcams FAILED: circuit-open` — 192
- `TFRs FAILED: circuit-open` — 192
- `Cached POIs bbox FAILED: circuit-open` — 195
- `LocalServerHealth: circuit OPEN ...` — 142

Confirmed cache-proxy is currently **not running on workstation** (nothing listening on port 4300), so this is operator-side state — but the app should not emit 700+ warnings just because its companion server is offline. The circuit-breaker pattern exists *specifically* to avoid hammering down servers with noise; we're doing exactly that in the log stream.

**Fix:** log only on circuit state transitions (OPEN → CLOSED, CLOSED → OPEN, HALF_OPEN probes). Every caller that hits an already-open circuit should get the fast-fail response silently (or at `D`).

**4. [LOW] `NarrationMgr: TTS ERROR: newspaper_1692_*` — 3 errors logged at `E`-level that are actually cancel events.**

All 3 occurrences (01:53:53, 02:43:14, 03:09:12) show the same pattern: a newspaper segment is mid-speech, a `HINT` segment gets enqueued, `NarrationMgr: cancelSegmentsWithTag(newspaper_1692)` kills the current segment, TTS callback fires `onError` (because speech was aborted, not because TTS genuinely failed), and `HistNewspaper` auto-advances. User experience is fine — the HINT plays, the newspaper resumes next tick — but the log says `E TTS ERROR`.

**Fix:** in the TTS `onError` listener, check if a cancel is in-flight and log at `I` (or suppress) instead of `E`. Distinguish "aborted by us" from "real TTS failure."

Related: **32 `cancelSegmentsWithTag` events** over 12 hours — all benign (HINT interrupts, user navigation). Worth checking one day whether that's the expected cancel rate.

**5. [LOW] WALK-SIM DWELL CAP hit (60s) resuming walk — 2 occurrences.**

Walk simulator has a 60-second max dwell. Both hits were during the 00:40-01:30 active-tour window. With Oracle tiles now running 2-3 minutes of narration, the 60s dwell cap may cut off real content. Revisit the cap value.
