### PoiCache singleton — every POI read in memory after one load

Operator: "we should cache everything POI out of the database, it doesn't change."

New `app-salem/src/main/java/com/example/wickedsalemwitchcitytour/content/PoiCache.kt` — Hilt `@Singleton`. On first `ensureLoaded()` call (Mutex-serialized), runs `salemPoiDao.findAll()` once and builds an immutable `Snapshot` with pre-built indexes:
- `byId: Map<String, SalemPoi>`
- `byCategory: Map<String, List<SalemPoi>>` (key uppercased, matches DAO semantics)
- `bySubcategory`, `byDistrict`
- `visible: List<SalemPoi>` (`defaultVisible == true`)
- `narrated: List<SalemPoi>` (mirrors the `is_narrated OR (haunt_sprite_id IS NOT NULL AND haunt_enabled)` AND visibility-gate SQL)
- `tour: List<SalemPoi>` (`isTourPoi`)
- `districts: List<String>` (sorted)

Read API mirrors `SalemPoiDao` 1:1. `findNearby` / `findVisibleNearby` / `findNarratedNearby` are in-memory lat/lng box filters + sorted by squared euclidean distance. `search` is in-memory string contains (lowercase, ordered by priority, default limit 50). `findNarratedInBbox`, `findByCategory`, etc. all hit pre-built indexes or filter the cached list.

Refactored consumers:
- `SalemContentRepository`: 16 POI read methods `getAllPois`, `getVisiblePois`, `getPoiById`, `getPoisByCategory`, `getPoisBySubcategory`, `getNarratedPois`, `getNarratedPoisByWave`, `getNarratedPoisInBbox`, `getPoisNearby`, `getNarratedPoisNearby`, `getVisiblePoisNearby`, `searchPois`, `getPoisByDistrict`, `getDistricts`, `getPoiCount`, `getVisiblePoiCount` all call `poiCache.ensureLoaded()` then route to the cache. `insertPois` (write path) still hits the DAO and calls `poiCache.invalidate()`.
- `FindViewModel`: removed direct `salemPoiDao` injection + private `allPoisCache` field. Now injects `PoiCache`, kicks off `ensureLoaded()` in `init`. `searchPoisByName`, `fetchPoiWebsiteDirectly`, `queryNearby` all use cache.

- `SalemMainActivity`: `loadHauntConfigsAsync` (S226) was direct-DAO-injected for one-shot startup load; repointed at PoiCache. `salemPoiDao` field replaced with `poiCache`.

### Full +10mi q=60 bake — parallel infrastructure

Operator: "I want everything 10 miles out of the BBOX to have full map coverage. Maps are cheap." Asked for q=60 to control bundle size. Asked for parallelization to speed the bake.

`render-tiles.js`: `BBOX_PLUS_10MI` for ALL zoom layers including z19 (replaces tiered shape). WebP `quality: 80` → `quality: 60`. Metadata version 3 → 4.

New parallel-bake infra:
- `tools/tile-bake/render-worker.js` — variant of render-tiles.js. Accepts env `WORKER_ID`, `NUM_WORKERS`, `WORKER_OUTPUT`, `WORKER_QUALITY`. Renders only blocks where `(bx * 100003 + by) % NUM_WORKERS === WORKER_ID` (100003 prime → even hash spread). Each worker writes its own mbtiles.
- `tools/tile-bake/bake-parallel.js` — coordinator. CLI: `node bake-parallel.js [num_workers] [quality]`. Spawns N child Node processes (each loads its own MapLibre context, ~500 MB RAM), waits for all, merges all worker mbtiles into `salem-custom.mbtiles`, cleans up worker outputs.

Bake result: **715,841 tiles in 547 seconds (9 min) via 8 workers** vs ~75 min projected single-threaded — ~8× speedup. Each worker did ~89,470 tiles / ~32 MB. Final `salem-custom.mbtiles` 253.9 MB.

q=60 + the new tiles being mostly suburbs/water/woods (not downtown) gave ~370 bytes/tile avg — well under my 700 MB projection.
