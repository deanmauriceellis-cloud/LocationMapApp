2. **Bug found + fixed (commit `e2270f4`).** Root cause: Express's default `qs` parser collapses `filter[route]=Red` into nested `{filter: {route: 'Red'}}`. `new URLSearchParams(req.query)` then renders that as `filter=%5Bobject+Object%5D`, MBTA ignores the malformed filter, returns the whole table. Fix: pass raw query string from `req.originalUrl` through verbatim, regex-strip any client-sent `api_key=...` segment to avoid double-keying, re-append cache-proxy's key server-side. Cache key also uses sanitized (key-stripped) string so hits don't depend on the secret. ### Post-close: CI failure follow-up (Wave 2 ci.yml regression)

Operator forwarded a GitHub Actions failure notification on commit `99f6741` (and re-run of `e2270f4`). Both runs failed at `:app-salem:verifyBundledAssets` with `Error: Cannot find module 'better-sqlite3'` — the Gradle task shells out to `cache-proxy/scripts/verify-bundled-assets.js` which requires `better-sqlite3`, and CI never installed cache-proxy's node_modules. `verifyBundledAssets` is wired into `preBuild.dependsOn` unconditionally (`app-salem/build.gradle:221`), so `-PskipPublishChain` doesn't skip it — only the publish-* tasks.

**Fix:** added a `Set up Node.js` step (`actions/setup-node@v4`, version 20, npm cache keyed on `cache-proxy/package-lock.json`) plus an `Install cache-proxy dependencies` step (`working-directory: cache-proxy`, `npm ci --no-audit --no-fund`) before the gradle invocations. CI now has the SQLite-reading deps the verify task needs, and the committed `salem_content.db` gets verified as part of every CI run.

Also added the explanatory comment that `verifyBundledAssets` still runs under `-PskipPublishChain` (only the publish-* tasks are skipped).
