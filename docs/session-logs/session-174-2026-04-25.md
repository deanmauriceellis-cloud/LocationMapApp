### Backend additions (admin-tours.js)

- `POST   /admin/salem/tours` — create tour. id + name required; sane defaults
  for everything else.
- `PATCH  /admin/salem/tours/:tour_id` — whitelist edit of name/theme/
  description/estimated_minutes/distance_km/difficulty/seasonal/icon_asset/
  sort_order. stop_count is NOT in the whitelist (auto-managed).
- `DELETE /admin/salem/tours/:tour_id` — hard delete in a transaction;
  cascades stops first, then deletes the tour row.
- `POST   /admin/salem/tours/:tour_id/stops` — add waypoint. Accepts either
  poi_id (POI-linked) or lat+lng (free) or both. If stop_order is supplied,
  shifts equal/greater stops down by 1; else appends. Echoes effective coords.
- `DELETE /admin/salem/tours/:tour_id/stops/:stop_id` — delete + renumber
  subsequent stops down by 1, all in a transaction.
- `POST   /admin/salem/tours/:tour_id/stops/reorder` — body `{order: [stop_id...]}`.
  Validates set is exactly the tour's current stop_id set, then rewrites
  stop_order 1..N in array order using a two-phase negate-then-rewrite to
  dodge any future unique-constraint windows.
- After every add/delete/reorder, a helper `syncStopCount` keeps
  `salem_tours.stop_count` in sync with the live row count.

Smoke-test: full CRUD round-trip exercised — create → patch metadata →
add free stop → add poi-linked stop → insert with shift → reorder → delete
middle (renumber) → delete tour. All passed; live DB reverted to the original
