## Phase 1 — Switch to byte-budget cache

Standard Android pattern: `Runtime.getRuntime().maxMemory() / 8` as the byte cap, with `sizeOf` returning each bitmap's `allocationByteCount`. Adapts automatically to device heap (low-end phones get a smaller cache, tablets/Lenovo get more headroom).

Defensive `Long` math: `(maxMemory / 8L).coerceAtMost(Int.MAX_VALUE.toLong()).toInt()` because `LruCache` takes an `Int` `maxSize` and `Runtime.maxMemory()` returns `Long`. Realistic Android heaps will never hit Int.MAX_VALUE (2 GiB), but the coerce guards against future-API-bump surprises and silences any static-analysis nit.

Diff:

```kotlin
// before
private val cache = object : LruCache<Long, Bitmap>(256) {
    override fun sizeOf(key: Long, value: Bitmap): Int = 1
}

// after
private val cache = object : LruCache<Long, Bitmap>(
    ((Runtime.getRuntime().maxMemory() / 8L).coerceAtMost(Int.MAX_VALUE.toLong())).toInt()
) {
    override fun sizeOf(key: Long, value: Bitmap): Int = value.allocationByteCount
}
```

Concrete budget on a typical device:
- Android default heap (no `largeHeap`): `maxMemory()` ≈ 256 MB → cache cap ≈ 32 MB ≈ 128 ARGB-8888 256×256 tiles.
- Lenovo TB305FU (8 GB RAM tablet, but per-app heap is still device-class-bound): same order of magnitude.

This is comfortably above any single-viewport tile count (~50 visible at typical zoom + prefetch ring) but well under the prior worst-case 64 MB ceiling, and — critically — it shrinks proportionally with whatever the OS hands the process.
