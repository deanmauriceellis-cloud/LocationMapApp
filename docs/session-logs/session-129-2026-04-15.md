### Step 5 ‚Äî On-device verification ‚úì

**First hurdle ‚Äî crash on open.** `SQLiteException: no such table: salem_witch_trials_articles`. The bundled `app-salem/src/main/assets/salem_content.db` was generated pre-Phase-9X and is missing the three new tables. Room's `fallbackToDestructiveMigration` didn't fire because the bundled `room_master_table.identity_hash` was empty, so Room treated the DB as "already migrated" and never created the new tables.

**Fix (bake the schema).** Used the generated `SalemContentDatabase_Impl.java` to extract the canonical `CREATE TABLE` SQL for the 3 witch-trials tables and the expected identity hash (`0545a31e8fb7ce05e575755b61532d46`), then patched the bundled DB with them + `VACUUM`. Schema matched, identity matched, tables empty.

**Second hurdle ‚Äî silent write loss.** With schema now valid, `articleDao.insertAll(16 rows)` returned without exception, but `articleDao.count()` immediately after (same coroutine) returned 0. Even `sqlite3` on the pulled device DB showed 0 rows. Our best guess: the prebuilt asset DB had been patched via `sqlite3` CLI in rollback-journal mode; on first open Room flipped `journal_mode=wal` and something about that transition left the new tables in a state where writes were accepted but never committed/checkpointed. Reproduced twice, across fresh `pm clear` installs.

**Fix (bundle the rows directly).** Wrote `cache-proxy/scripts/bundle-witch-trials-into-db.js` (requires `better-sqlite3`) that copies the 16 rows from PG straight into the bundled `salem_content.db`, wraps the writes in a transaction, and `VACUUM`s afterwards. Now the asset DB ships with 16 rows pre-populated; the Repository's asset-JSON fallback path is kept as a safety net but normally becomes a no-op (`count()==16` short-circuits hydrate).
