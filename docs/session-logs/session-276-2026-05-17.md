### Filter-and-Map bug diagnosis + search-mode containment + tour pause/resume

**Operator field test feedback (Lenovo, post-synonyms install):** *"filter and map doesn't filter and map - all the pois are still visible / clicking POI should clear the filter"*. Then: *"put verbose debugging on all of this when we are in debug mode, why guess?"*

**Approach:** Per `feedback_diagnose_before_changing.md` + `feedback_verbose_debug_until_production.md` — instrumented the entire filter-and-map flow with `DebugLogger.d("FilterMap", ...)` lines (R8-strippable in release), reinstalled on Lenovo, captured trace. Operator-pacing favored "log first, fix second" so the fix wasn't a guess.

**Logging coverage added:**
- `clearAllPoiMarkers` / `clearPoiMarkers(layerId)` — per-layer counts before clear + state of filter flags.
- `replaceAllPoiMarkers(places)` — count + state pre-clear.

- `enterFilterAndMapMode` — entry banner with job-active flags + `overlays.size(pre)`.

- `exitFilterAndMapMode` — entry banner + pre-state.
- `loadCachedPoisForVisibleArea` — bbox + caller stack frame (so we know which path triggered it).
- Marker click handler (filter-map markers) — POI tap.

**Diagnostic trace from Lenovo run of "pot" query (33 results):**
```
01:41:23.502 ENTER mode label='"pot"' results=33 cachePoiJob.active=false ... overlays.size(pre)=631
01:41:23.511 clearAllPoiMarkers() removing 0 markers across 0 layers []
01:41:23.548 ENTER mode added 33 filter-map markers. overlays.size(post-add)=664
```

**Root cause exposed by trace:** the visible POIs on the map are NOT tracked in the `poiMarkers` layer map that `clearAllPoiMarkers()` walks. They live in a second cache-backed system: `narrationMarkers` / `narrationMarkerCache` / `narrationMarkersOnOverlay` (S118/S234, `SalemMainActivity.kt:2594-2796`). Two parallel marker systems, only one wired into Filter-and-Map. Result: 631 narration markers stay on the map, filter adds 33 on top → 664 total → "filter didn't filter."

**Operator design direction (S276):** Search is its own mode. Tour pauses when search opens; auto-resumes if user backs out of search; stays paused with a "Resume" chip if user picks a result. Chip lives in the status line (persistent), tap to resume.
