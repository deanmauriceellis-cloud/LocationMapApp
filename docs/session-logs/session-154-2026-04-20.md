## Third pass — GPS cursor freeze on Lenovo (map didn't move while walking)

Operator reported: "the gps isn't moving my position on the map when I walk, I don't seem to be moving." Pulled 5.8 MB of live log from Lenovo HNY0CY0W and traced.

### Root cause

The Lenovo TB305FU's `TYPE_SIGNIFICANT_MOTION` sensor **never fires**. Today's log shows:

```
D/STATIONARY: freeze — stationary (never fired (305s since start), events=0)
```

The MotionTracker reports `events=0` even after 5+ minutes of real walking. Consequence: `MotionTracker.isStationary()` stays true indefinitely, and the `stationaryFrozen` gate in SalemMainActivity suppresses the visible-layer updates (marker, polyline, trigger rings). The only escape paths were:

1. `update.speedMps > 0.5f` — LocationManager-provided speed. Often null / 0 on this device.
2. `distFromLast > 25m` — the distance hatch. Fires only after the user has physically walked 25m from the last accepted fix.

With both paths failing for slow walking, the cursor only updated in ~25m hops. Hence the "I'm not moving" feel.

### Fix

Added a **derived-speed escape hatch** in the `stationaryFrozen` block: distance/time between consecutive fixes. If ≥ 0.3 m/s (≈ 0.7 mph), unfreeze.

- Computes `derivedMps = distFromLast / dtSeconds` using a new `lastGpsPointMs` timestamp.
- At 1 Hz fix rate + 1.4 m/s walking pace, every fix unfreezes → cursor tracks each step naturally.
- Slow/stationary users still get the 25m distance hatch as a safety net.
- Zero impact on the legitimate desk-jitter freeze (speed near zero → derived-speed near zero → stays frozen).

### Files changed

- `app-salem/src/main/java/.../ui/SalemMainActivity.kt`:
  - Added `internal var lastGpsPointMs: Long = 0L` alongside `lastGpsPoint`.
  - Added derived-speed check inside `val stationaryFrozen = run { ... }`.
  - Updated `lastGpsPointMs = System.currentTimeMillis()` adjacent to `lastGpsPoint = point`.
