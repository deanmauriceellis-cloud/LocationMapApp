### Inverted-pan fix â€” manual perspective touch transform

Operator field-test: drag-down made the map go up (and vice-versa) in tilt mode. Direction was inverted, not just imprecise.

**Root cause:** `MotionEvent.transform(android.graphics.Matrix)` does NOT correctly handle perspective matrices on Android (verified empirically â€” comments in some AOSP versions imply it strips or mis-applies the bottom row of the homography). With `invMatrix` from our top-pinched perspective, the broken transform inverted the y direction. Confirmed by stripping the transform call entirely: pan direction was correct (drag-down â†’ map content moved down with finger), albeit not pixel-perfect under perspective compression.

**Fix:** rebuild the MotionEvent manually using `Matrix.mapPoints` (which DOES handle perspective correctly). For each pointer, copy the `PointerProperties` and `PointerCoords`, run the (x, y) through `invMatrix.mapPoints`, then construct a fresh event via the multi-pointer `MotionEvent.obtain` overload. This preserves the touched-POI-follows-finger feel under perspective AND keeps direction correct.

### Real wedge regression â€” `setMapCenterOffset` broke tile-fetch bounds

Operator caught the smoking gun: route polylines were rendering THROUGH dark areas in the wedges (`/tmp/now.png` â€” blue route extends past where tiles end). Overlays drawing where tiles aren't = the projection knows about those screen pixels, but the tile-fetch viewport doesn't cover them.

**Root cause:** `MapView.setMapCenterOffset(0, +extra/2)` shifts osmdroid's projection center by `+extra/2` pixels, but the tile-fetch bbox is computed as `center Â± half_view_height`. With our taller MapView (`height = containerH + extra`), `half_view = (containerH+extra)/2`. Visible tile range becomes `[center.y - half, center.y + half] = [extra/2, containerH + 3*extra/2]` in MapView local pixels. The MapView's actual rendering range is `[0, containerH+extra]`. The intersection leaves `[0, extra/2)` of MapView pixels with NO tiles â€” the dark band the operator saw.

**Fix:** SYMMETRIC extension â€” `extra/2` above + `extra/2` below the natural TiltContainer bounds. MapView's geometric center stays at TiltContainer center = operator's GPS-rendered position, so `setMapCenterOffset` is unnecessary. Tile-fetch covers the full MapView naturally. Default View pivot also lines up with the operator, so the FAB x1..x5 magnify pivots correctly without explicit `pivotX/Y` (those are removed). The bottom half of the extension extends below the screen and is clipped by `CoordinatorLayout` (parent default `clipChildren=true`) â€” no visual cost.
