### Iteration 2 — duration perception fix

Operator: "skeleton lasted less than 300ms and disappeared." Diagnosed: original code multiplied `scale = rawScale * bell.coerceAtLeast(0.4f)` — the alpha bell-curve also shrank the sprite at start/end of the dance, so only the middle "big + bright" bobs registered (~300ms). Fix: decoupled scale from alpha. Alpha now uses a flat-top trapezoid (15% fade-in / 70% full alpha / 15% fade-out); scale is purely the depth bob, so size stays vivid throughout. Reduced bob amplitude range 0.3-0.7 → 0.15-0.35; tightened scale clamp 0.55-1.8 (was 0.25-2.0). Added per-fire logcat `fire <poiId> sprite=… duration=…ms amp=… cycles=…` so future iterations have ground truth for what the device actually used.

### Iteration 3 — 16-frame split + spinning behavior (then reverted)

S170 sprite WebPs are 16-frame animated heading sequences (frame 0 = facing east per `~/AI-Studio/map-sprites/generate-angles.py` convention). My initial loader only used frame 0, so the skeleton always faced east no matter where you stood. Wrote a Python one-shot that splits each animated WebP into per-frame WebPs at `assets/sprites/<id>/00.webp` … `15.webp` (~2.2 MB total for 7 sprites × 16 frames). SpriteOverlay loader switched from single Bitmap cache to `Array<Bitmap?>` per sprite, with first-fire random start frame + random direction + 8-14 fps cycling so the sprite "spins" through all 16 angles during the dance.

### Iteration 5 — minute-scale duration + elliptical motion path + gentle bob

Operator: "It only lasted about .5 seconds for me to watch and I set it to 3 minute and it should not be pulsating, it should be gentle changes to aspect" — three asks at once. Three changes:

Sprite floats ~50 px above the POI marker.

3. **Gentle aspect change.** Bob amplitude dropped 0.15-0.35 → 0.06-0.14; scale clamp tightened 0.55-1.8 → 0.7-1.4. No more pulsating.

Operator immediately rejected: "we can't take guesses at this, we need to align this. it isn't about the skeletons - they align at the right starting angle. ### Iteration 7 — actual root cause: osmdroid heading-up canvas rotation

Read `WickedAnimationOverlay.kt` line 22-27 docstring: when osmdroid `MapView.getMapOrientation` is non-zero (heading-up tour mode), the canvas is pre-rotated by the device compass heading before our overlay draws. Polygons (water, fireflies, grass) are rotationally symmetric so they look fine; sprites with an obvious up-axis appear rotated by however much the map is rotated. The 150° was the device compass heading at the time, not a sprite-render bug.

**The actual fix (3 lines in one place):**

- `CameraState` gained `mapOrientationDeg: Float = 0f`.
- `WickedAnimationOverlay.draw()` now passes `mapView.mapOrientation` into the camera each frame.
- `SpriteOverlay.draw()`: before drawing each sprite, if `camera.mapOrientationDeg != 0f` then `canvas.save() / canvas.rotate(-mapOrientationDeg, cx, cy) / drawBitmap / restore()`. Sprite stays anchored to the POI's projected screen position, but always renders upright on screen regardless of map heading.

The earlier per-sprite rotation hack (`SPRITE_ROTATION_DEG`) was deleted along with `rotationDeg` on `ActiveSwoop`. Operator confirmed: "Much better."
